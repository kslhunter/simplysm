# Feature 1.1 SdDropdown destroy 시 popup 정리

## 참조 자료

### 버그 증상

- sd-data-sheet 내 `@if`로 sd-select를 조건부 렌더링
- 편집 모드 → sd-select dropdown open → 항목 클릭 시 focus가 popup으로 이동
- 셀 blur → 편집 모드 종료 → `@if` false → SdDropdownControl destroy
- `document.body`에 append된 popup이 제거되지 않고 orphaned 상태로 잔류
- 이후 수동 클릭에도 popup이 사라지지 않음

### 수정 대상 파일

- `packages/sd-angular/src/ui/overlay/dropdown/sd-dropdown.control.ts`

### 근본 원인

`SdDropdownControl`의 `$effect`는 `open` signal이 false로 변경될 때만 else 분기(popup 제거)를 실행한다.
`@if`로 컴포넌트가 destroy될 때는 signal 변경 없이 effect가 torn down되어 cleanup이 실행되지 않는다.
popup은 `document.body`에 있으므로 Angular의 일반 DOM 정리 대상도 아니다.

### 수정 내용 (이미 구현됨)

`ngOnDestroy` 추가 + `_popupEl` 필드 캐싱:
```typescript
private _popupEl?: HTMLElement;

// $effect: open=true 시 this._popupEl 에 캐싱, open=false 시 this._popupEl = undefined
// ngOnDestroy: this._popupEl?.remove()
```

### 설계 결정

| # | 결정사항 | 선택 | 근거 |
|---|---------|------|------|
| D1 | destroy 시 cleanup 방식 | ngOnDestroy + _popupEl 캐싱 | 기존 focus 관리 로직 보존, destroy 타이밍에 contentChild signal 접근 불필요 |
| D2 | $effect onCleanup 사용 여부 | 미사용 | onCleanup은 effect 재실행 전에도 호출되어 정상 close 시 focus 복귀 로직이 무효화됨 |
| D3 | 테스트 환경 | vitest + Playwright browser 모드 | Angular TestBed는 DOM 환경 필요, Playwright browser provider가 실제 브라우저 제공 |
| D4 | $effect else 브랜치에서 popupEl 참조 방식 | _popupEl 캐시 사용 (popupElRef() 미사용) | Angular의 change detection 순서상 effects가 content queries 초기화(refreshContentQueries) 전에 실행됨. contentChild.required()를 else 브랜치에서 호출하면 초기 렌더링 시 NG0951 에러 발생. _popupEl 캐시를 사용하면 open=false 초기 상태에서 안전하게 null check 후 skip 가능 |
| D5 | vitest browser provider 패키지 | @vitest/browser-playwright | vitest 4.x에서 provider를 문자열 "playwright"로 지정하는 방식이 deprecated됨. @vitest/browser-playwright 패키지의 playwright() 팩토리 함수를 사용해야 함 |
| D6 | Angular AOT 컴파일 | @analogjs/vite-plugin-angular (jit: true) | vitest + vite에서 Angular 컴포넌트를 올바르게 컴파일하려면 Angular 전용 vite 플러그인 필요. TypeScript 5.8과의 호환성 문제(TS AST 구조 변경)로 인해 @angular/compiler-cli 패치 필요 |

## 구현계획

### 배경

`SdDropdownControl`은 `document.body.appendChild()`로 popup을 DOM에 직접 이동시킨다.
이 방식 때문에 Angular의 컴포넌트 destroy 시 Angular DOM 정리로 popup이 제거되지 않는다.
`ngOnDestroy` + `_popupEl` 캐싱으로 수정 완료.
이제 이 동작을 검증하는 자동화 테스트를 작성한다 — Angular TestBed + vitest Playwright browser 모드 사용.

### 목표

- `SdDropdownControl` destroy 시 popup 제거 자동화 테스트
- vitest Playwright browser 모드로 sd-angular 테스트 환경 구축

### 비목표

- `SdDropdownControl`의 다른 기능(scroll, keydown, mouseover 등) 전면 테스트
- jsdom 기반 테스트 (실제 DOM focus 동작 검증 불가)

#### 테스트 구조

`packages/sd-angular/tests/sd-dropdown.destroy.spec.ts`:

```typescript
// TestHostComponent: show signal로 @if를 통한 조건부 렌더링
// open signal 없이 SdDropdownControl 인스턴스에 직접 접근 ([(open)] two-way binding 불필요)

describe("SdDropdownControl", () => {
  afterEach(() => { TestBed.resetTestingModule(); });

    it("open=false로 변경 시 popup이 document.body에서 제거된다");

  });
});
```

## 요구명세

```gherkin
Feature: 1.1 SdDropdown destroy 시 popup 정리

  Background:
    Given SdDropdownControl이 @if 조건부 렌더링 컨테이너 안에 있다
    And dropdown이 open 상태이다 (popup이 document.body에 append된 상태)

  Rule: 컴포넌트 destroy 시 open된 popup을 document.body에서 제거한다

    Scenario: @if false로 컴포넌트 destroy 시 popup 제거
      Given SdDropdownControl의 open이 true이다
      When @if 조건이 false가 되어 SdDropdownControl이 destroy된다
      Then document.body에 dropdown popup 엘리먼트가 존재하지 않는다

  Rule: 정상 close 경로의 기존 동작을 유지한다

    Scenario: open signal이 false로 변경 시 popup 제거 및 스타일 초기화
      Given SdDropdownControl의 open이 true이다
      When open이 false로 변경된다
      Then popup이 document.body에서 제거된다
      And popup의 position/opacity/transform 스타일이 초기화된다

    Scenario: popup에 focus가 있을 때 close 시 focus 복귀
      Given SdDropdownControl의 open이 true이다
      And popup 내부 요소에 focus가 있다
      When open이 false로 변경된다
      Then focus가 SdDropdownControl host 요소로 복귀한다
```
