# 디버그: Judge FAIL 시 Main이 원인 정보 없이 잘못된 개선안을 생성

## 에러 증상

- **에러 메시지:** (런타임 에러 아님 — 행동 결함)
- **위치:** `.claude/skills/sd-prompt/SKILL.md` Step 4-4 (Judge subagent) ~ Step 4-6 (개선 루프)
- **재현:** sd-prompt에서 Eval 실행 → Judge가 FAIL 보고 → Main이 원인을 이상하게 추측하여 엉뚱한 개선안 제시

## 근본 원인 추적 (Why Chain)

1. **증상:** Main이 FAIL 원인을 이상하게 추측하여 엉뚱한 개선안을 제시
2. **왜?** → Main이 eval 대상이 실제로 무엇을 했는지 모른 채 원인을 추측
3. **왜?** → Main이 Judge의 간략한 판정 근거(테이블 한 줄)만 보고, 실제 eval 실행 출력(`run-output.json`)이나 workspace 산출물을 읽지 않음
4. **왜?** → Judge → Main 간 정보 전달 형식이 `| 항목 | 판정 | 근거(한 줄) |`로 제한됨. Step 4-6(1)에도 1차 데이터를 읽으라는 지시 없음
5. **근본 원인:** **Judge가 보유한 풍부한 컨텍스트(run-output.json, workspace 파일, eval 대상의 행동 흐름)가 Main에게 전달되지 않는 구조적 정보 단절**

### 정보 단절 구조

```
Judge (풍부한 컨텍스트)          Main (빈약한 컨텍스트)
├── run-output.json 전문    →   | 항목 | 판정 | 근거(한 줄) |
├── workspace 파일 전체     →   (전달 안 됨)
└── eval 대상의 행동 흐름   →   (전달 안 됨)
                                 ↓
                            FAIL 원인 추측 (정보 부족)
                                 ↓
                            엉뚱한 개선안
```

### 논의 과정에서 검토한 대안과 기각 사유

| 대안 | 기각 사유 |
|------|-----------|
| Judge가 판정 + 원인 분석 + 개선안 생성(1~3)을 전부 수행 | 개선안은 **전체 시나리오의 FAIL을 통합**해야 하므로, 시나리오별 독립 Judge가 내면 충돌/누락 발생 |
| Main이 run-output.json을 직접 읽기 | stream-json 형식으로 시나리오당 수천 줄 → 여러 시나리오 시 context window 폭발, 가독성 낮음 |
| Main이 Judge 없이 전부 수행 | 위와 동일한 context 부담 + 시나리오 병렬 처리 불가 |

## 해결 방안

### 방안 A: Judge 역할 확대 (판정 + 원인 분석 + 실제 행동 인용) + Main이 통합 개선안 생성

- **설명:** 정보가 있는 쪽(Judge)이 분석까지 수행하고, 통합이 필요한 작업(개선안)은 Main이 담당한다.

- **역할 재분배:**

  ```
  Judge (시나리오별, 병렬)          Main (전체 통합)
  ├── 판정 (PASS/FAIL)              ├── 모든 Judge 보고서 수신
  ├── FAIL 원인 판별 (A/B)          ├── Contrastive 분석 (2)
  └── 기대 vs 실제 행동 인용        └── 통합 개선안 생성 (3)
  ```

- **변경 포인트:**

  **Step 4-4 Judge prompt 변경:**
  ```markdown
  <!-- Before -->
  각 체크리스트 항목에 대해:
  1. PASS 또는 FAIL을 판정하라
  2. 판정의 근거(evidence)를 구체적으로 기술하라

  결과를 아래 형식으로 출력하라:

  | 항목 | 판정 | 근거 |
  |------|------|------|
  | {항목} | PASS/FAIL | {근거} |

  통과율: N/M

  <!-- After -->
  각 체크리스트 항목에 대해:
  1. PASS 또는 FAIL을 판정하라
  2. 판정의 근거(evidence)를 구체적으로 기술하라

  결과를 아래 형식으로 출력하라:

  | 항목 | 판정 | 근거 |
  |------|------|------|
  | {항목} | PASS/FAIL | {근거} |

  통과율: N/M

  FAIL 항목이 있으면 아래를 추가로 출력하라:

  ### FAIL 상세 분석
  각 FAIL 항목에 대해:
  - **항목:** {체크리스트 항목}
  - **기대:** {체크리스트가 요구한 행동}
  - **실제:** {eval 대상이 실제로 한 행동 — run-output.json에서 관련 부분을 인용}
  - **원인 판별:** (A) 프롬프트가 부족/불명확 또는 (B) Eval 체크리스트가 부정확
  - **판별 근거:** {왜 A 또는 B로 판단했는지}
  ```

  **Step 4-6 (1) 변경:**
  ```markdown
  <!-- Before -->
  #### (1) FAIL 원인 판별
  FAIL 항목을 분석하여 원인을 판별한다: (A) 프롬프트가 부족 또는 (B) Eval 체크리스트가 부정확

  <!-- After -->
  #### (1) FAIL 원인 확인
  Judge 보고서의 "FAIL 상세 분석"에서 각 FAIL 항목의 원인 판별(A/B)과 실제 행동 인용을 확인한다.
  Judge의 판별에 동의하지 않으면 재판별하되, Judge가 인용한 "실제 행동"을 근거로 판단한다.
  ```

- **장점:**
  - Judge가 이미 읽은 run-output.json의 컨텍스트를 최대한 활용 — 추가 비용 없음
  - "기대 vs 실제" 인용으로 Main이 정확한 개선안을 생성할 수 있음
  - 시나리오별 병렬 Judge → 속도 유지
  - 개선안은 Main이 전체 시나리오를 보고 통합 생성 → 시나리오 간 충돌 방지

- **반론:**
  - Judge 역할이 "판정자"에서 "판정자 + 분석가"로 확대 — subagent 복잡도 증가
  - Judge의 원인 판별이 틀릴 수 있음 (단, Main이 재판별 가능하고 "실제 행동 인용"이 있으므로 완화됨)
  - Judge 출력이 길어져 Agent tool 반환 크기 제한에 걸릴 가능성

- **점수:**
  - 근본성: **9**/10 — 정보 단절의 근본 원인을 직접 해결하면서 통합 개선안의 정합성도 보장
  - 실용성: **8**/10 — Judge prompt + Step 4-6 (1) 수정으로 구현 가능. Main context 부담 최소
  - 견고성: **8**/10 — Judge 판별이 틀려도 "실제 행동 인용"으로 Main이 재판별 가능
  - → **평균 8.3/10**

### 방안 B: 수행 안 함

- **설명:** 현재 상태를 유지한다.

- **장점:**
  - 변경 비용 없음
  - 기존 동작에 대한 regression 위험 없음

- **반론:**
  - Main이 FAIL 원인을 계속 잘못 추측하여 개선 루프가 수렴하지 않는 문제가 지속
  - 사용자가 매번 수동으로 원인을 교정해야 하는 부담

- **점수:**
  - 근본성: **1**/10 — 문제 미해결
  - 실용성: **10**/10 — 변경 없음
  - 견고성: **3**/10 — 문제 지속으로 스킬 신뢰도 하락
  - → **평균 4.7/10**

## 추천

**방안 A** (평균 8.3/10)

정보가 있는 쪽(Judge)이 원인 분석과 실제 행동 인용까지 수행하고, 통합이 필요한 작업(Contrastive 분석, 개선안 생성)은 전체 시나리오를 조망하는 Main이 담당한다. 정보 충실도와 통합 정합성을 모두 확보하는 구조.
