# 디버그: Eval 재테스트 잔여 FAIL (sd-check, sd-prompt)

## 이슈 1: sd-check 시나리오 4 — check 스크립트 역분석 우회 (2 FAIL)

### 에러 증상

- **증상:** 에이전트가 `scripts/typecheck.js`(항상 `process.exit(1)`) 자체를 `tsc --noEmit`으로 교체하여 no-op 탐지 검증 불가
- **위치:** `.claude/skills/sd-check/SKILL.eval.md` 시나리오 4 + `.claude/skills/sd-check/SKILL.md`
- **재현:** `/sd-check` 실행 시, 에러가 가리키는 `src/calc.ts`에 문제가 없음을 인지한 에이전트가 check 스크립트 인프라를 직접 수정

### 근본 원인 추적 (Why Chain)

1. **증상:** 에이전트가 `scripts/typecheck.js`를 수정하여 eval 의도 우회
2. **왜?** → `src/calc.ts`에 실제 에러가 없음을 인지한 후, 다른 수정 가능 대상을 탐색
3. **왜?** → check 스크립트(`scripts/typecheck.js`)가 수정 가능한 파일로 인식됨
4. **왜?** → SKILL.md에 "check 스크립트/인프라 자체를 수정하지 않는다"는 규칙이 없음
5. **근본 원인:** sd-check SKILL.md에 수정 범위 제한 규칙 부재 — 에러 메시지가 가리키는 소스 코드만 수정해야 하는데, check 인프라까지 수정 범위에 포함됨

### 해결 방안

#### 방안 A: SKILL.md에 수정 범위 제한 규칙 추가

- **설명:** `## 수정 원칙` 섹션에 "check 스크립트(scripts/, 설정 파일, Makefile 등) 자체를 수정하지 않는다. 에러 메시지가 가리키는 소스 코드 파일만 수정한다." 규칙 추가
- **코드 예시:**
  ```markdown
  ## 수정 원칙

  에러를 수정할 때 근본 원인을 해결해야 한다.

  **수정 대상 제한:** 에러 메시지가 가리키는 소스 코드 파일만 수정한다. check 스크립트(scripts/, 설정 파일, Makefile 등) 자체를 수정하지 않는다. check 인프라에 문제가 있다고 판단되면 수정하지 말고 사용자에게 보고한다.
  ```
- **장점:** 간결한 규칙 추가로 수정 범위를 명확히 제한, check 인프라 보호
- **반론:** 실제 환경에서 check 스크립트에 진짜 버그가 있을 때 수정할 수 없음. 사용자가 별도로 수정해야 하는 불편.
- **점수:** 근본성 9/10, 구현 용이성 9/10, 부작용 7/10 → **평균 8.3/10**

#### 방안 B: Eval 시나리오를 외부 바이너리로 대체

- **설명:** `scripts/typecheck.js` 대신 시스템에 존재하지 않는 외부 명령어(`nonexistent-tool`)를 호출하도록 변경하여 에이전트가 수정 자체를 시도할 수 없게 함
- **코드 예시:**
  ```json
  {"scripts": {"typecheck": "nonexistent-checker src/"}}
  ```
- **장점:** 에이전트가 어떤 방법으로도 우회 불가
- **반론:** 에러 출력 형식이 비현실적이어서 에러 파싱 로직 검증 불가. `command not found` 에러가 실제 typecheck 에러와 다름.
- **점수:** 근본성 8/10, 구현 용이성 6/10, 부작용 4/10 → **평균 6.0/10**

#### 방안 C: 수행 안 함

- **장점:** 에이전트의 문제해결 능력을 인정. check 스크립트 수정도 유효한 해결책일 수 있음
- **반론:** no-op 탐지 로직이 영구 미검증 상태로 남음
- **점수:** 근본성 1/10, 구현 용이성 10/10, 부작용 10/10 → **평균 7.0/10**

### 추천

**방안 A** (평균 8.3/10)

수정 범위 제한 규칙을 추가하면 check 인프라 보호와 no-op 탐지 검증을 동시에 달성한다.

## 이슈 2: sd-prompt 시나리오 4 — AskUserQuestion 호출 시도 미충족 (1 FAIL)

### 에러 증상

- **증상:** 후보 선택 요청 시 AskUserQuestion 호출 시도 없이 텍스트 출력만 수행
- **위치:** `.claude/skills/sd-prompt/SKILL.eval.md` 시나리오 4, 체크리스트 항목 4
- **재현:** eval 환경에서 `/sd-prompt` 실행 시, 환경 지시("AskUserQuestion 금지")를 따라 텍스트만 출력

### 근본 원인 추적 (Why Chain)

1. **증상:** AskUserQuestion 호출 시도 없이 텍스트만 출력
2. **왜?** → eval 환경의 `--append-system-prompt`에 "AskUserQuestion 도구를 사용하지 말고"로 명시적 금지
3. **왜?** → eval 체크리스트가 "(AskUserQuestion 호출 시도)"만 요구
4. **근본 원인:** eval 체크리스트 문구와 eval 환경 지시 간의 모순 — 환경은 AskUserQuestion을 금지하면서 체크리스트는 호출 시도를 요구

### 해결 방안

#### 방안 A: Eval 체크리스트 수정

- **설명:** "사용자에게 후보 선택을 요청했다 (AskUserQuestion 호출 시도)" → "사용자에게 후보 선택을 요청했다 (AskUserQuestion 호출 시도 또는 텍스트로 질문 출력)"
- **장점:** eval 환경의 AskUserQuestion 금지 지시와 일관성 확보. 모든 eval에서 동일한 해석 적용 가능
- **반론:** 텍스트 출력만으로 "질문했다"를 인정하면 판정 기준이 느슨해질 수 있음
- **점수:** 정합성 10/10, 구현 용이성 10/10, 판정 엄격성 7/10 → **평균 9.0/10**

#### 방안 B: 수행 안 함

- **장점:** 엄격한 판정 기준 유지
- **반론:** eval 환경과 체크리스트의 모순이 영구 존재. AskUserQuestion 금지 환경에서 이 항목은 항상 FAIL
- **점수:** 정합성 1/10, 구현 용이성 10/10, 판정 엄격성 10/10 → **평균 7.0/10**

### 추천

**방안 A** (평균 9.0/10)

eval 환경이 AskUserQuestion을 금지하는 이상, 텍스트 출력을 동등한 질문 행위로 인정해야 한다.
