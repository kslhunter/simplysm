- `Could not resolve "D : /workspaces-12/oscom/node_modules/core-js/modules/es.symbol.description.js"` (268개 모듈 + abortcontroller-polyfill + resize-observer-polyfill)
- 위치: `packages/sd-cli/src/pkg-builders/client/SdPolyfillPlugin.ts`

- [debug.md](./debug.md)

### 관련 파일
- `packages/sd-cli/src/pkg-builders/client/SdPolyfillPlugin.ts` — 수정 대상
- `packages/sd-core-node/src/utils/PathUtils.ts` — `PathUtils.posix()` (수정 불필요, 사용 제거)

## 요구명세

Feature: 1.1 SdPolyfillPlugin esbuild resolve 수정

  Background:
    Given sd-cli가 Angular 클라이언트 패키지를 빌드한다
    And SdPolyfillPlugin이 "Chrome >= 61" browserslist 쿼리로 등록되어 있다

  Rule: 가상 polyfill 모듈의 import가 OS 무관하게 resolve되어야 한다

    Scenario: Windows에서 core-js 모듈이 정상 resolve된다
      Given core-js-compat이 268개 모듈을 반환한다
      When esbuild가 sd-polyfill namespace의 가상 모듈을 로드한다
      Then 모든 core-js import가 "Could not resolve" 없이 번들링된다

    Scenario: abortcontroller-polyfill이 정상 resolve된다
      Given abortcontroller-polyfill이 sd-cli의 node_modules에 설치되어 있다
      When esbuild가 sd-polyfill namespace의 가상 모듈을 로드한다
      Then abortcontroller-polyfill import가 정상 resolve된다

    Scenario: resize-observer-polyfill이 정상 resolve된다
      Given resize-observer-polyfill이 sd-cli의 node_modules에 설치되어 있다
      When esbuild가 sd-polyfill namespace의 가상 모듈을 로드한다
      Then resize-observer-polyfill import가 정상 resolve된다

  Rule: core-js-compat 기반 동적 polyfill 목록이 유지되어야 한다

    Scenario: browserslistQuery에 따라 필요한 polyfill만 포함한다
      Given browserslistQuery가 "Chrome >= 61"이다
      When core-js-compat이 polyfill 목록을 생성한다
      Then Chrome 61에서 지원하지 않는 ES feature에 대한 polyfill만 포함된다

    Scenario: 존재하지 않는 모듈은 skip한다
      Given core-js-compat이 polyfill 불가 항목(WeakRef 등)을 목록에 포함한다
      When _require.resolve()가 해당 모듈을 찾지 못한다
      Then 해당 모듈을 무시하고 나머지 모듈로 가상 모듈을 생성한다

  Rule: polyfill 모듈은 sd-cli의 의존성에서 resolve한다

    Scenario: 소비 프로젝트에 core-js가 없어도 동작한다
      Given 소비 프로젝트의 node_modules에 core-js가 없다
      And sd-cli의 dependencies에 core-js가 있다
      When SdPolyfillPlugin이 polyfill 가상 모듈을 생성한다
      Then sd-cli의 core-js에서 모듈이 resolve된다

## 구현계획

### 배경

v12.16.38에서 `SdPolyfillPlugin`을 도입하여 정적 `lib/chrome61-polyfills.js`를 `core-js-compat` 기반 동적 생성으로 리팩터링했다. 이때 `_require.resolve()`로 얻은 절대 경로를 `PathUtils.posix()`로 변환하여 import문에 삽입했으나, esbuild 커스텀 namespace에서 `resolveDir` 없이는 파일시스템 import가 resolve되지 않는다.

### 목표

- `onLoad` return에 `resolveDir` 추가하여 esbuild가 커스텀 namespace 가상 모듈의 import를 resolve 가능하게 함
- 절대 경로 대신 bare specifier를 사용하여 OS 무관한 경로 해석

### 비목표

- `PathUtils.posix()` 자체 수정 (다른 곳에서도 사용되므로 수정 불필요)
- polyfill 목록 변경 (기존 `core-js-compat` 로직 유지)

### 설계

#### 변경 사항 (`SdPolyfillPlugin.ts`)

1. **import 변경:** `PathUtils` import 제거, `fileURLToPath`(url), `path` import 추가
2. **bare specifier 사용:** `import "${PathUtils.posix(absPath)}"` → `import "core-js/modules/${mod}.js"` (abortcontroller-polyfill, resize-observer-polyfill도 동일)
3. **`resolveDir` 추가:** `onLoad` return에 `resolveDir: path.dirname(fileURLToPath(import.meta.url))` 추가
4. **`_require.resolve()` 유지:** try/catch 존재 여부 확인용으로만 사용, 반환값은 사용하지 않음

#### `resolveDir` 선택 근거

`path.dirname(fileURLToPath(import.meta.url))`는 sd-cli 패키지의 소스/dist 디렉터리를 가리킨다. 이 디렉터리에서 `node_modules` 트리를 타고 올라가면 sd-cli의 dependencies(`core-js`, `abortcontroller-polyfill`, `resize-observer-polyfill`)를 찾을 수 있다. `_require`도 동일한 resolution context를 사용하므로, `_require.resolve()`가 성공하면 esbuild도 동일 bare specifier로 resolve 가능하다.

### 대안 검토

| 접근 방식 | 선택 여부 | 이유 |
|-----------|-----------|------|
| bare specifier + resolveDir | 채택 | esbuild 커스텀 namespace의 표준 패턴, OS 무관 |
| 절대 경로 + resolveDir | 미채택 | Windows 드라이브 문자가 URL scheme으로 오해석될 위험 잔존 |
| 이전 방식(정적 파일) 롤백 | 미채택 | core-js-compat 기반 동적 생성의 이점(browserslist 연동) 상실 |
