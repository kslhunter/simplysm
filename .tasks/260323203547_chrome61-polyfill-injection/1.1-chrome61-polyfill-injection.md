# Feature 1.1 Chrome 61 Polyfill 자동 주입

## 참조 자료

### 현재 빌드 구성
- `SdNgBundler.ts`: Angular 클라이언트 빌드, `browserslist(["Chrome >= 61"])` 타겟
- `SdStyleBundler.ts`: 스타일 번들링, 동일 브라우저 타겟
- esbuild `inject`로 `node-stdlib-browser` shim 이미 주입 중 (`SdNgBundler.ts:551-555`)
- 클라이언트 패키지에 `src/polyfills.ts`가 있으면 esbuild entryPoint로 자동 추가 (`SdNgBundler.ts:491-495`)

### TypeScript 설정
- `tsconfig.base.json`: target ES2022, lib ES2021
- 패키지별 tsconfig에서 `lib: ["ES2021", "DOM"]` 확장

### Chrome 61에서 누락된 ES API (ES2018~ES2022)
- ES2018: `Promise.prototype.finally`
- ES2019: `Array.flat/flatMap`, `Object.fromEntries`, `String.trimStart/trimEnd`
- ES2020: `globalThis`, `String.matchAll`, `Promise.allSettled`
- ES2021: `Promise.any`, `AggregateError`, `String.replaceAll`, `WeakRef`, `FinalizationRegistry`
- ES2022: `Array.at`, `String.at`, `Object.hasOwn`, `Error.cause`

### Chrome 61에서 누락된 Web API
- `ResizeObserver` (Chrome 64), `AbortController` (Chrome 66)
- `queueMicrotask` (Chrome 71), `structuredClone` (Chrome 98)

### 제약사항
- publish되는 라이브러리(sd-core-browser 등)에 polyfill bare import를 넣으면 소비 프로젝트에서 resolve 실패
- `resize-observer-polyfill` 메인 엔트리의 `declare global`이 DOM 내장 타입과 충돌
- polyfill은 빌드 도구(sd-cli)의 책임

## 요구명세

```gherkin
Feature: 1.1 Chrome 61 Polyfill 자동 주입

  Background:
    Given sd-cli로 Angular 클라이언트를 빌드한다
    And 빌드 타겟이 Chrome >= 61이다

  Rule: sd-cli가 ES API polyfill을 자동 주입한다

    Scenario: ES2018~ES2022 API가 Chrome 61에서 사용 가능하다
      When Angular 클라이언트를 빌드한다
      Then 번들에 core-js 기반 ES API polyfill이 포함된다
      And Chrome 61에서 Promise.finally, Array.flat, globalThis, Promise.any, Array.at 등이 정상 동작한다

    Scenario: 최신 Chrome에서는 polyfill이 실행되지 않는다
      Given 최신 Chrome 브라우저에서 실행한다
      When polyfill 코드가 로드된다
      Then core-js의 feature detection으로 이미 지원하는 기능은 skip한다

  Rule: sd-cli가 Web API polyfill을 자동 주입한다

    Scenario: Chrome 61에서 누락된 Web API가 사용 가능하다
      When Angular 클라이언트를 빌드한다
      Then 번들에 ResizeObserver, AbortController, queueMicrotask, structuredClone polyfill이 포함된다
      And Chrome 61에서 해당 Web API가 정상 동작한다

  Rule: polyfill은 빌드 도구에만 존재하며 publish되는 라이브러리에 포함되지 않는다

    Scenario: sd-core-browser에 polyfill 코드가 없다
      When sd-core-browser 라이브러리를 빌드한다
      Then dist에 polyfill import가 포함되지 않는다
      And package.json에 core-js, resize-observer-polyfill, abortcontroller-polyfill 의존성이 없다

    Scenario: 소비 프로젝트에서 resolve 에러가 발생하지 않는다
      Given 외부 프로젝트에서 @simplysm/sd-core-browser를 사용한다
      When 해당 프로젝트를 빌드한다
      Then polyfill 관련 "Could not resolve" 에러가 발생하지 않는다

  Rule: 기존 빌드 동작과 호환된다

    Scenario: node-stdlib-browser shim이 기존처럼 주입된다
      When Angular 클라이언트를 빌드한다
      Then node-stdlib-browser shim이 기존과 동일하게 inject된다
      And polyfill과 shim이 충돌하지 않는다
```

## 구현계획

### 배경

`SdNgBundler`에서 `browserslist(["Chrome >= 61"])`로 타겟을 설정하지만, esbuild는 문법만 다운레벨링하고 런타임 API polyfill은 하지 않는다. 현재 `lib/cordova-entry.js`를 별도 entry point로 추가하는 선례가 있으며, `node-stdlib-browser` shim을 esbuild `inject`로 주입하는 패턴도 존재한다.

### 목표

- Chrome 61에서 ES2018~ES2022 API와 Web API(ResizeObserver, AbortController, queueMicrotask, structuredClone) polyfill 자동 주입
- 빌드 도구(sd-cli)에서 처리하여 publish 라이브러리에 polyfill 코드가 포함되지 않도록 함

### 비목표

- browserslist 쿼리에 따른 동적 polyfill 결정 (고정 목록 사용)
- ES2023 이상 polyfill

### 설계

#### 파일 구조

```
packages/sd-cli/
  lib/
    chrome61-polyfills.js    # 신규 — esbuild entry point용 plain JS
  package.json               # core-js, resize-observer-polyfill, abortcontroller-polyfill 추가
  src/pkg-builders/client/
    SdNgBundler.ts           # sd-polyfills entry point 추가
```

#### chrome61-polyfills.js 내용

core-js selective import + Web API polyfill. plain JS이므로 타입 충돌 없음.

```javascript
// ES2018~ES2022
import "core-js/actual/promise/finally";
import "core-js/actual/array/flat";
import "core-js/actual/array/flat-map";
// ... (전체 목록은 기존 chrome61-polyfills.ts와 동일)

// Web APIs
import "core-js/actual/queue-microtask";
import "core-js/actual/structured-clone";
import "abortcontroller-polyfill/dist/abortcontroller-polyfill-only";

// ResizeObserver
import ResizeObserver from "resize-observer-polyfill";
if (typeof window !== "undefined" && !("ResizeObserver" in window)) {
  window.ResizeObserver = ResizeObserver;
}
```

#### SdNgBundler 변경

`entryPoints`에 `sd-polyfills` 추가, `IndexHtmlGenerator.entrypoints`에 `["sd-polyfills", true]`를 `polyfills` 앞에 추가.

```typescript
entryPoints: {
  "sd-polyfills": path.resolve(
    path.dirname(fileURLToPath(import.meta.url)),
    "../../../lib/chrome61-polyfills.js",
  ),
  main: this._mainFilePath,
  ...(polyfills entry 기존 로직 유지),
},

// IndexHtmlGenerator
entrypoints: [
  ["sd-polyfills", true],
  ["polyfills", true],
  ["styles", false],
  ["main", true],
],
```

### 대안 검토

| 접근 방식 | 선택 | 이유 |
|-----------|------|------|
| esbuild `inject`로 주입 | 미채택 | inject는 identifier replacement용, side-effect-only polyfill에 부적합 (tree-shake 위험) |
| 클라이언트 `src/polyfills.ts`에 직접 작성 | 미채택 | 프로젝트마다 중복 관리 필요 |
| publish 라이브러리(sd-core-browser)에 포함 | 미채택 | 소비 프로젝트에서 transitive dep resolve 실패 |
| 별도 entry point (`sd-polyfills`) | 채택 | `cordova-entry` 선례, IndexHtmlGenerator와 자연스럽게 통합 |
