# sd-review: 코드 리뷰 및 이슈 도출

이 스킬의 목적은 지정 경로의 코드를 분석하여 문제점/개선점을 도출하고, `/sd-plan-dev`가 바로 실행할 수 있는 plan.md를 생성하는 것이다.

코드 자체가 "무엇이 문제인지"를 이미 담고 있으므로, spec.md(요구 명세서) 단계를 거치지 않고 plan.md를 직접 생성한다.

한국어로 응답한다.

## 프로세스 맥락

이 스킬은 코드 분석 → 구현의 2단계 파이프라인이다:
1. **`/sd-review`** (이 스킬) → 코드 분석, 이슈 도출, plan.md 생성
2. `/sd-plan-dev` → plan.md 기반 구현

## AskUserQuestion 사용 규칙

리뷰의 명확화는 최소한이어야 한다. 질문이 많으면 리뷰 자체가 노이즈가 된다.

**질문 기준**: "이 답을 알면 이슈 판정이 달라지는가?" — 달라지는 경우에만 질문한다.

질문이 여러 개일 때 **한 번에 하나씩** 순차적으로 진행한다:
1. 질문에 대한 설명을 출력한다.
2. `---`를 출력한다.
3. AskUserQuestion을 호출한다.
4. 답변을 반영하여 다음 질문으로 넘어간다.

## 입력

`$ARGUMENTS`로 분석할 경로를 받는다. 파일, 디렉토리, 또는 복수 경로를 공백으로 구분하여 지정한다.

## 프로세스

### 1단계: 대상 파일 수집

`$ARGUMENTS`를 공백으로 분리하여 각 경로를 처리한다:

- **파일 경로**: 해당 파일을 직접 분석 대상에 추가
- **디렉토리 경로**: Glob으로 하위의 코드 파일을 수집한다. 대상 확장자: `*.ts`, `*.js`, `*.tsx`, `*.jsx`, `*.py`, `*.java`, `*.go`, `*.rs`, `*.cs`, `*.kt`, `*.swift`, `*.rb`, `*.php`, `*.vue`, `*.svelte`

여러 경로가 지정되면 모든 파일을 하나의 대상 목록으로 통합한다.

수집된 파일이 없으면 "분석할 코드 파일이 없습니다" 메시지를 표시하고 종료한다.

### 2단계: 프로젝트 컨텍스트 수집 및 리뷰 명확화

분석의 정확도를 높이기 위해, Multi-Pass 분석 **전에** 프로젝트 맥락을 확보한다.

#### 자동 수집

프로젝트 루트에서 다음 파일을 탐색하여 자동으로 컨텍스트를 수집한다:
- **lint 설정**: `.eslintrc*`, `biome.json`, `.prettierrc*` 등 — 프로젝트의 코드 스타일/규칙
- **컴파일 설정**: `tsconfig.json`, `pyproject.toml` 등 — strict 모드, 타겟 환경
- **프로젝트 문서**: `CLAUDE.md`, `README.md` — 프로젝트 컨벤션, 아키텍처 설명
- **기존 테스트**: 테스트 파일 패턴 — 테스트 프레임워크, 테스트 컨벤션

#### 리뷰 명확화

자동 수집으로 해소되지 않는 **진단 불확실성**이 있을 때만 AskUserQuestion으로 질문한다.

명확화 관점 (아래 "리뷰 명확화 관점" 참조):
1. 코드 맥락/의도 — 의도적 패턴 vs 실수
2. 운영 환경/규모 — 성능 이슈의 심각도 판단 기준
3. 리뷰 범위 — 집중할 것 / 무시할 것
4. 기존 수용된 문제(Baseline) — 이미 알고 있는 기술 부채
5. 프로젝트 컨벤션 — 자동 수집으로 파악 안 되는 암묵적 규칙

자동 수집 결과로 충분히 판단 가능하면 질문 없이 바로 다음 단계로 진행한다.

### 3단계: Multi-Pass 분석

4개 분석 Agent를 **병렬**로 spawning한다. 각 Agent에게 대상 파일 목록, 역할별 분석 지침(아래 "분석 패스 정의" 참조), 그리고 **2단계에서 수집한 프로젝트 컨텍스트**를 전달한다.

각 Agent의 분석 결과는 자유 형식으로 수집한다 (구조화는 4단계에서 수행).

Agent 프롬프트 구조:

```
다음 파일들을 **{패스명}** 관점에서 분석하여 문제점과 개선점을 도출하라.

## 대상 파일
{파일 목록 — 각 파일을 Read로 읽어 분석}

## 프로젝트 컨텍스트
{2단계에서 수집한 lint 설정, 컴파일 설정, 프로젝트 컨벤션, 명확화 결과}

## 분석 관점
{해당 패스의 분석 지침}

## 출력 형식
각 이슈에 대해 다음을 기술:
- **위치**: 파일 경로 및 관련 코드 영역 (함수명, 라인 범위 등)
- **문제**: 현재 무엇이 잘못되었거나 개선 가능한지
- **기대 상태**: 수정 후 어떻게 동작해야 하는지
- **심각도 제안**: Critical / Major / Minor / Trivial

이슈가 없으면 "발견된 이슈 없음"을 보고한다.
```

### 4단계: 결과 수집 및 이슈 분류

4개 Agent의 결과를 수집하여:

1. **중복 병합**: 다른 패스에서 동일 코드를 다른 관점으로 지적한 경우 하나로 병합한다.
2. **분류 부여**: 각 이슈에 심각도/카테고리/우선순위를 부여한다 (아래 "이슈 분류 체계" 참조).
3. **정렬**: 우선순위 내림차순으로 정렬한다.

이슈가 없으면 "발견된 이슈가 없습니다" 메시지만 표시하고 종료한다.

### 5단계: plan.md 작성

1. Bash로 `date +%y%m%d%H%M%S`를 실행하여 타임스탬프를 얻는다.
2. topic은 분석 대상 경로에서 핵심 키워드를 추출하여 영문 kebab-case로 만든다 (예: `src/services/` → `review-services`).
3. `.tasks/{timestamp}_{topic}/` 디렉토리를 생성한다.
4. 아래 "plan.md 작성 가이드"에 따라 plan.md를 작성한다.
5. **정합성 검증**: 4단계에서 분류된 모든 이슈가 plan.md의 Slices와 Verification List에 빠짐없이 반영되었는지 확인한다. 누락이 있으면 추가한다.

## 분석 패스 정의

### 보안 패스

다음 관점에서 보안 취약점을 찾는다:
- **Injection**: SQL injection, command injection, XSS 등 사용자 입력이 검증 없이 실행/출력되는 코드
- **인증/인가**: 인증 우회 가능성, 불충분한 권한 검증, 세션 관리 문제
- **시크릿 노출**: 하드코딩된 비밀번호, API 키, 토큰, 연결 문자열
- **입력 검증**: 미검증 사용자 입력, 부적절한 타입 캐스팅, 경로 순회

### 성능 패스

다음 관점에서 성능 문제를 찾는다:
- **알고리즘 복잡도**: O(n²) 이상의 중첩 루프, 비효율적 탐색/정렬
- **N+1 문제**: 루프 내 반복적 DB/API 호출
- **불필요 할당**: 반복문 내 불필요한 객체 생성, 대량 배열 복사
- **캐싱 누락**: 동일 입력에 대한 반복 계산, 자주 조회되지만 캐싱되지 않는 데이터

### 설계/아키텍처 패스

다음 관점에서 설계 문제를 찾는다:
- **SOLID 위반**: 단일 책임 원칙(거대 클래스/함수), 개방-폐쇄(switch/if 체인), 의존성 역전(구체 클래스 직접 참조)
- **Code Smell**: God Class(과도한 줄 수/메서드 수), Feature Envy(다른 클래스 데이터를 과도하게 사용), Data Clump(반복되는 파라미터 그룹)
- **의존성 문제**: 순환 의존성, 부적절한 레이어 간 참조, 과도한 결합
- **구조 문제**: 중복 코드, 죽은 코드, 일관성 없는 패턴

### 에러처리/엣지케이스 패스

다음 관점에서 에러 처리 문제를 찾는다:
- **Null/Undefined**: null 체크 없는 반환값 사용, optional 필드 미처리
- **리소스 관리**: 닫히지 않는 연결/스트림/핸들, finally 블록 누락
- **에러 전파**: 무시되는 에러(빈 catch), 부적절한 에러 메시지, 에러 타입 미구분
- **경계값**: 빈 배열/문자열, 음수, 오버플로우, 동시 접근

## 이슈 분류 체계

### 심각도

| 레벨 | 점수 | 기준 |
|------|------|------|
| Critical | 4 | 크래시, 데이터 손실, 보안 취약점, 우회 불가 |
| Major | 3 | 핵심 기능 저하, 우회 어려움 |
| Minor | 2 | 부차 기능 영향, 우회 가능 |
| Trivial | 1 | 외관/스타일, 기능 영향 없음 |

### 카테고리

Bug / Security / Performance / Maintainability / Reliability / Architecture / Testing / Documentation

### 우선순위

```
Priority = 심각도(1~4) × 영향범위(1~3) × 수정용이성(1~3)
```

- 영향범위: System-wide=3, Module=2, File=1
- 수정용이성: Quick fix=3, Medium=2, Large refactor=1

## 리뷰 명확화 관점

아래 5가지 관점에서 진단 불확실성이 있는지 확인한다. 자동 수집으로 해소되면 질문하지 않는다.

### 1. 코드 맥락/의도

코드가 의도적 선택인지 실수인지에 따라 이슈 판정이 달라진다.
- 자동 수집: 코드 주석, 커밋 메시지, README의 설계 설명
- 질문 예시: "이 서비스가 단일 클래스로 큰 것이 의도적 설계(Facade)인가요?"

### 2. 운영 환경/규모

동일 코드도 규모에 따라 문제 여부가 달라진다. O(n²)이 n=10이면 무관하고 n=100만이면 Critical이다.
- 자동 수집: 배포 설정, 인프라 파일, 성능 테스트
- 질문 예시: "이 API의 예상 동시 요청 수는?"

### 3. 리뷰 범위

사용자가 관심 있는 영역에 집중하고 나머지는 노이즈를 줄인다.
- 질문 예시: "특별히 집중해서 봐야 할 관심사가 있나요?", "무시해도 되는 부분이 있나요?"

### 4. 기존 수용된 문제 (Baseline)

이미 알고 있지만 수용 중인 기술 부채를 다시 지적하면 노이즈다.
- 자동 수집: TODO/FIXME 주석, 이슈 트래커 연동
- 질문 예시: "이 코드베이스에서 이미 알고 있는 기술 부채가 있나요?"

### 5. 프로젝트 컨벤션

자동 수집된 lint/tsconfig 외에 팀만 아는 암묵적 규칙이 있을 수 있다.
- 자동 수집: lint 설정, editorconfig, 기존 코드 패턴
- 질문 예시: "이 프로젝트에서 따르는 특별한 코딩 규칙이 있나요?"

## plan.md 작성 가이드

`.claude/skills/sd-plan/template-plan.md` 형식을 따른다.

### Spec 섹션

리뷰 출처를 기록한다:

```
## Spec

- Source: sd-review 분석
- Target: {분석 대상 경로}
```

### Contracts 섹션

수정 후 기대되는 코드 구조를 정의한다:
- 변경되는 인터페이스/타입/함수 시그니처
- 기존 코드의 패턴과 컨벤션을 따름
- 기존 프로젝트의 재사용 가능한 모듈을 최대한 활용

### Slices 섹션

발견된 이슈를 독립 작업 단위로 분해한다:
- 각 Slice는 하나의 Agent가 독립 구현 가능한 단위
- Files가 겹치는 Slice는 Depends on으로 순차 실행 강제
- 우선순위가 높은 이슈(Critical/Major)를 먼저 배치

### Verification List 섹션

각 이슈의 수정을 **동작으로 검증**하는 항목을 정의한다:
- "이 입력을 주면 이 동작을 하는가" 형태
- 단순한 것부터 복잡한 것 순서로 정렬
- 각 항목에 해당 Slice 번호를 매핑
