# 계획 수립 템플릿

작업 유형별 계획 템플릿과 Todo 작성 가이드.

## 표준 계획 출력 형식

```markdown
## 분석 결과

### 요청 요약
[사용자 요청 1-2문장 요약]

### 코드베이스 분석

**관련 파일**:
- `path/to/file1.ts` - [역할]
- `path/to/file2.ts` - [역할]

**기존 패턴**:
- [패턴 1]: [설명]
- [패턴 2]: [설명]

**재사용 가능 코드**:
- `functionName` in `file.ts:42` - [용도]

**주의사항**:
- [제약사항 1]
- [제약사항 2]

### 선택된 접근방식

**옵션**: [접근방식 이름]

**선택 이유**:
- [이유 1]
- [이유 2]

**Trade-off**:
- 장점: [장점]
- 단점: [단점]

---

## 구현 계획

### 1단계: [단계명]
**목표**: [달성할 것]

- [ ] 세부 작업 1
- [ ] 세부 작업 2

**수정 파일**:
- `file1.ts` - [변경 내용]

**주의사항**: [주의할 점]

---

### 2단계: [단계명]
...

---

## 검증 계획
- [ ] [테스트/검증 항목 1]
- [ ] [테스트/검증 항목 2]

---

## 다음 단계
계획 확정 시 "구현 시작" 호출.
```

## 작업 유형별 템플릿

### 새 기능 추가

```markdown
## 구현 계획: [기능명]

### 1단계: 기반 구조 생성
- [ ] 새 파일/모듈 생성
- [ ] 타입/인터페이스 정의
- [ ] 기본 구조 설정

### 2단계: 핵심 로직 구현
- [ ] 주요 기능 구현
- [ ] 비즈니스 로직 작성
- [ ] 데이터 처리 로직

### 3단계: 통합
- [ ] 기존 코드와 연결
- [ ] 진입점 설정
- [ ] 설정/환경 변수 추가

### 4단계: 테스트
- [ ] 단위 테스트 작성
- [ ] 통합 테스트 (필요시)
- [ ] 수동 테스트

### 5단계: 마무리
- [ ] 코드 정리
- [ ] 문서화 (필요시)
```

### 버그 수정

```markdown
## 버그 수정 계획: [버그 설명]

### 1단계: 문제 확인
- [ ] 버그 재현 확인
- [ ] 근본 원인 파악
- [ ] 영향 범위 확인

### 2단계: 수정 구현
- [ ] 수정 코드 작성
- [ ] 관련 코드 조정

### 3단계: 검증
- [ ] 버그 해결 확인
- [ ] 회귀 테스트
- [ ] 엣지 케이스 확인

### 4단계: 완료
- [ ] 코드 정리
- [ ] 테스트 추가 (재발 방지)
```

### 리팩토링

```markdown
## 리팩토링 계획: [대상 영역]

### 1단계: 준비
- [ ] 현재 테스트 확인/보강
- [ ] 리팩토링 범위 확정
- [ ] 롤백 계획 수립

### 2단계: 구조 변경
- [ ] 새 구조 생성
- [ ] 코드 이동/재구성
- [ ] 인터페이스 조정

### 3단계: 기존 코드 정리
- [ ] 사용처 업데이트
- [ ] 기존 코드 제거
- [ ] import 정리

### 4단계: 검증
- [ ] 전체 테스트 실행
- [ ] 기능 동작 확인
- [ ] 성능 영향 확인
```

### 설정/환경 변경

```markdown
## 설정 변경 계획: [변경 내용]

### 1단계: 영향 분석
- [ ] 영향 받는 환경 확인
- [ ] 의존성 확인
- [ ] 하위 호환성 확인

### 2단계: 변경 구현
- [ ] 설정 파일 수정
- [ ] 코드 조정 (필요시)
- [ ] 환경 변수 업데이트

### 3단계: 검증
- [ ] 각 환경에서 테스트
- [ ] CI/CD 파이프라인 확인
```

## Todo 작성 가이드

### 좋은 Todo 항목

```markdown
- [ ] UserService에 validateEmail() 메서드 추가
- [ ] LoginForm 컴포넌트에서 이메일 검증 호출
- [ ] 검증 실패 시 에러 메시지 표시 로직 추가
```

### 나쁜 Todo 항목

```markdown
- [ ] 이메일 검증 기능 만들기  # 너무 모호함
- [ ] 코드 작성  # 구체적이지 않음
- [ ] 테스트  # 무엇을 테스트하는지 불명확
```

### Todo 작성 원칙

| 원칙 | 설명 |
|------|------|
| 구체적 | 무엇을 어디에 하는지 명시 |
| 측정 가능 | 완료 여부 판단 가능 |
| 독립적 | 가능한 개별적으로 완료 가능 |
| 적절한 크기 | 15분-2시간 작업 단위 |

## 복잡도 판단 기준

| 복잡도 | 특징 | 접근 |
|--------|------|------|
| 단순 | 단일 파일, 명확한 변경, 기존 패턴 적용 | Phase 축소 |
| 중간 | 2-5개 파일, 새 로직 추가, 기존 패턴 확장 | 표준 프로세스 |
| 높음 | 5개+ 파일, 아키텍처 영향, 새 패턴 도입 | 심층 분석 |

## 마일스톤 분할

대규모 작업 시 마일스톤으로 분할:

```markdown
## 프로젝트: [대규모 기능명]

### 마일스톤 1: 기반 구축
- 핵심 데이터 모델 설계
- 기본 API 구조 생성
- 데이터베이스 스키마

### 마일스톤 2: 핵심 기능
- 주요 비즈니스 로직
- API 엔드포인트 구현
- 기본 테스트

### 마일스톤 3: UI/UX
- 프론트엔드 컴포넌트
- 사용자 인터페이스
- 통합 테스트

### 마일스톤 4: 완성
- 엣지 케이스 처리
- 성능 최적화
- 문서화 및 배포
```

각 마일스톤은 독립적으로 배포/검증 가능해야 함.
