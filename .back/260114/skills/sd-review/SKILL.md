# 코드 리뷰

패키지 또는 디렉토리 단위의 종합적인 코드 리뷰를 위한 체계적인 워크플로우 제공.

## 리뷰 대상

사용자가 명시하지 않으면 확인:

| 유형 | 대상 |
|-----|------|
| 패키지 | 특정 패키지 |
| 디렉토리 | 특정 디렉토리 |
| 파일 | 특정 파일 |
| 전체 | 프로젝트 전체 |

## 리뷰 워크플로우

### 1단계: 범위 파악

리뷰 대상의 구조와 범위 파악:
- 디렉토리 구조 확인
- 파일 목록 및 개수 파악
- 주요 진입점 식별

### 2단계: 프로젝트 컨텍스트 확인

프로젝트별 가이드라인 확인:
- `CLAUDE.md` 또는 유사한 프로젝트 규칙 문서
- `eslint.config.ts`, `tsconfig.json` 등 설정 파일
- 기존 코드 패턴 및 컨벤션

### 3단계: 종합 리뷰 수행

1. `sd-review/references/review-checklist.md` 읽기
2. sd-code-reviewer 에이전트 호출 시 체크리스트를 프롬프트에 포함
3. 에이전트가 체크리스트 기준으로 종합 리뷰 수행

### 4단계: 이슈 검증

1. sd-code-reviewer가 반환한 이슈 목록을 sd-review-validator 에이전트에 전달
2. 에이전트가 각 이슈의 컨텍스트를 확인하고 오탐 필터링
3. 정제된 최종 이슈 목록 수신

### 5단계: 결과 제시

검증된 리뷰 결과를 아래 형식에 맞춰 사용자에게 보고.

## 심각도 분류

| 등급 | 기준 | 예시 |
|-----|------|------|
| Critical | 즉시 수정 | 보안 취약점, 데이터 손실 가능성, 주요 기능 오작동 |
| Major | 수정 권장 | 버그 발생 가능성, 성능 병목, 유지보수 어려움 |
| Minor | 선택적 개선 | 스타일 불일치, 작은 최적화, 문서화 개선 |

## 리뷰 결과 형식

```markdown
# 코드 리뷰 결과

## 리뷰 대상
- 범위: {패키지명 또는 디렉토리}
- 파일 수: N개

## 검증 요약
- 원본 이슈: N건
- 유효한 이슈: N건
- 오탐 필터링: N건
- 보류: N건

## 유효한 이슈 (수정 필요)

### Critical

#### [C1] {이슈 제목}
- **파일**: `path/to/file.ts:123`
- **설명**: {상세 설명}
- **수정 제안**: {구체적인 수정 방법}

### Major
...

### Minor
...

## 오탐 (수정 불필요)

### [F1] {이슈 제목}
- **파일**: `path/to/file.ts:123`
- **오탐 사유**: {왜 수정이 불필요한지}

## 보류 (추가 확인 필요)

### [D1] {이슈 제목}
- **파일**: `path/to/file.ts:123`
- **확인 필요 사항**: {사용자가 판단해야 할 내용}

## 총평
{전반적인 코드 품질 평가}
```
