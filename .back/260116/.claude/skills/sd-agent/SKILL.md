# 프로젝트를 위한 에이전트 개발

프로젝트의 `.claude/agents/` 디렉토리에 "특정 전문성과 성격을 가진 자율 전문가 에이전트"를 작성하는 지침 제공.

## 워크플로우

| 단계 | 작성/수정 | 검토만 |
|------|----------|--------|
| 0. 요청 이해 | ✓ | ✓ |
| 1. 가이드 분석 | ✓ | ✓ |
| 2. 계획 수립 | ✓ | - |
| 3. 검증 | 계획 검증 | 대상 파일 검증 |
| 4. 타당성 검토 | ✓ | ✓ |
| 5. 실행 | 작성/수정 | - |
| 6. 최종 결과 | 작업요약 | 검토결과 |

### 0. 요청 이해

사용자 요청을 분석하여 **작업 유형** 파악:
- **작성**: 새 에이전트 생성
- **수정**: 기존 에이전트 변경
- **검토**: 기존 에이전트 검증만 수행

불명확한 경우 질문:
- 어떤 전문가가 필요한가? (예: 아키텍트, 분석가, 리뷰어)
- 어떤 성격/태도를 가져야 하는가? (예: 확신 있는, 꼼꼼한, 깊이 있는)
- 언제 이 에이전트를 호출해야 하는가?

요청이 명확하면 바로 다음 단계로 진행.

### 1. 가이드 분석

- 작성 가이드 참고: `sd-agent/references/writing-guide.md`
- 기존 에이전트를 모범 사례로써 분석: `.claude/agents/*`
- (수정/검토 시) 대상 에이전트 파일 분석

### 2. 계획 수립 (작성/수정 시)

> 검토만 하는 경우 이 단계 건너뜀

다음 항목을 포함한 구체적 계획 준비:
- Frontmatter 내용 (name, description)
- 에이전트의 역할과 전문성
- 핵심 원칙
- 입력/출력 형식
- 접근 방식

계획이 준비되면 사용자에게 간략히 제시.

### 3. 검증

**작성/수정 시**: 2단계 계획을 검증
**검토 시**: 대상 에이전트 파일을 검증

`sd-instruction-reviewer` 에이전트 호출:
```
가이드: sd-agent/references/writing-guide.md
대상: (계획 내용 또는 에이전트 파일)
```

### 4. 타당성 검토

3단계 검증 결과에서 **버그성 문제만 필터링**하여 제외:

| 제외 대상 | 설명 | 예시 |
|----------|------|------|
| 사실 오류 | 지적 내용이 사실과 다름 | "A가 없다"고 지적했는데 실제로 A가 존재함 |
| 잘못된 참조 | 존재하지 않는 파일/위치 참조 | 없는 파일에 대한 지적 |
| 가이드 오해 | 가이드를 잘못 해석함 | 선택적 항목을 필수로 오해 |
| 이미 해결됨 | 지적 사항이 이미 반영되어 있음 | "X를 추가해야 함" 근데 X가 이미 있음 |
| 범위 벗어남 | 검증 대상과 무관한 지적 | 다른 파일에 대한 수정 요구 |

위 표에 해당하는 지적만 제외하고, 스타일/표현 개선 제안은 유지.

**결과물 활용**:
- **작성/수정 시**: 유효한 피드백을 2단계 계획에 반영하여 계획 업데이트
- **검토 시**: 유효한 피드백을 6단계 검토 결과로 제시

### 5. 실행 (작성/수정 시)

> 검토만 하는 경우 이 단계 건너뜀

4단계에서 업데이트된 계획을 기반으로 실제 파일 작성/수정.

### 6. 최종 결과

**작성/수정 시 - 작업 요약**:
- 생성/수정된 파일 목록
- 주요 작성 내용
- 적용된 구조 및 패턴

**검토 시 - 검토 결과**:
- 4단계에서 필터링된 최종 개선 항목 제시
- 개선 항목이 없으면 "개선 항목 없음" 메시지만 출력
- 형식:
  ```
  ## 개선 항목

  **1. <문제 제목(파일명:라인번호)>**
  - **문제**: <현재 상태 설명>
  - **제안**: <구체적 수정 방안>
  ```
