### 0. 요청 이해

**목표**: 작업 유형과 범위 파악

사용자 요청을 분석하여 **작업 유형** 파악:
- **신규 기능**: 새로운 기능 추가
- **기능 수정**: 기존 기능 변경/개선
- **버그 수정**: 오류/결함 해결
- **리팩토링**: 구조 개선 (동작 변경 없음)

불명확한 경우 질문:
- 어떤 패키지에서 작업해야 하는가?
- 기대하는 동작은 무엇인가?
- 관련된 기존 코드나 이슈가 있는가?

요청이 명확하면 바로 다음 단계로 진행.

### 3. 계획 수립

**목표**: 구체적 구현 계획 작성

다음 형식으로 계획 작성:

```markdown
## 작업 개요
- **작업 유형**: (신규 기능 / 기능 수정 / 버그 수정 / 리팩토링)
- **영향 패키지**:
- **요약**:

## 파일 변경 목록
| 파일 | 작업 | 설명 |
|------|------|------|
| `path/to/file.ts` | 생성/수정/삭제 | 변경 내용 |

## 구현 순서
1. ...

## 테스트 전략
- **단위 테스트**:
- **통합 테스트**: (필요/불필요)
```

### 4. 계획 검증

**목표**: 계획의 품질 검토

Task 도구로 `sd-dev-plan-reviewer` 에이전트 호출:
```
가이드: .claude/references/dev-guide.md
대상: (계획 내용)
```

검토 관점:
- 아키텍처 일관성
- 테스트 커버리지
- 의존성 분석
- 호환성

### 5. 타당성 검토

**목표**: 검증 결과에서 버그성 오류 필터링 후 반복 여부 결정

3단계 검증 결과에서 **버그성 문제만 필터링**하여 제외:

| 제외 대상 | 설명 |
|----------|------|
| 사실 오류 | 지적 내용이 사실과 다름 |
| 잘못된 참조 | 존재하지 않는 파일/위치 참조 |
| 가이드 오해 | 가이드를 잘못 해석함 |
| 이미 해결됨 | 지적 사항이 이미 반영되어 있음 |
| 범위 벗어남 | 검증 대상과 무관한 지적 |

**반복 여부 결정**:
- 유효한 문제점 있음 → **2단계(계획 수립)로 돌아가서 수정 후 재검증**
- 유효한 문제점 없음 → 5단계(구현)로 진행

**주의사항:**
- 플랜모드에서 레포팅시 제외대상은 표시하지 않음.

### 7. 코드 검증

**목표**: 영향받은 패키지/통합테스트 검증

**영향받은 패키지만** 대상으로 검증.

**주의사항:** 패키지 수정시, 프로젝트 전체에 대한 타입체크와 린트를 임의로 수행하지 않음 (통합테스트는 영향을 받는 테스트만 수행)

실패 시 코드 수정 후 재검증.

### 8. 최종 결과

**목표**: 작업 결과 요약

- 생성/수정된 파일 목록
- 주요 구현 내용
- 테스트 결과
- 추가 고려사항 (문서화, 배포 등)
