# 검토 스킬

프로젝트/패키지의 문제점 및 개선점을 종합 검토.

## 작업 흐름

### 1. 범위 결정

- 명시 없음 → 전체 프로젝트
- 패키지명 명시 → 해당 패키지

### 2. 문제점/개선점 검토

대상을 분석하여 문제점/개선점을 검토한다.

**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

**코드 검토:**

- 에러 처리, 엣지 케이스 누락, 타입 안전성 등 코드 품질 전반
- 직접 구현 대신 표준 기능 또는 외부 npm 라이브러리로 대체 가능 여부
- simplysm 패키지 기능 우선 사용 권장 (필요시 의존성 추가)
- 코딩 지침(TYPESCRIPT_RULES.md) 준수 여부
- 프로젝트 내 유사 구현이 있으면 재사용 권장
- deprecated API 사용 여부
- 성능 최적화 가능 여부

**아키텍처 검토:**

- 패키지 간 의존성 구조 적절성
- 순환 의존성 여부
- 계층 구조 및 책임 분리
- 확장성 및 유지보수성

**테스트 검토:**

- 기존 테스트가 실제 기능을 충분히 커버하는지 확인
- 누락된 엣지 케이스나 시나리오에 대한 테스트 추가 필요 여부
- 단위테스트와 통합테스트 모두 확인

**문서화 검토:**

- 공개 API에 대한 TSDoc 존재 여부
- README.md 내용의 정확성 및 최신성
- 복잡한 로직에 대한 주석 적절성
- 문서화 위치: 코드 관련 → TSDoc (우선) / 구조적·설계 관련 → README.md
- 코드관련 주석중 사용자가 알아야 할것은 반드시 TSDoc에 작성되어야함. 사용자는 몰라도 되는 내용은 일반 주석으로 작성

### 3. 결과 재검토 (제외/문서화)

지적을 재검토(제외/문서화) 한다.

**중요 원칙:**

- **보수적 제외**: 확실한 제외 근거가 있을 때만 제외
- 애매하거나 제외사유가 약한 경우 → 유효한 지적으로 유지
- 해당 코드가 **왜 그렇게 작성되었는지** 파악하라. 의도를 설명하는 근거(주석, TSDoc, README, 테스트 등)가 있다면 **인용하라**. 근거를 찾았다면 → 제외 대상

**다음에 해당할 때만 제외:**

| 기준      | 설명                                                    |
|---------|-------------------------------------------------------|
| 의도된 동작  | TSDoc, 주석, README.md, 테스트, 명명 규칙 등으로 **명확한 증거** 확인 가능 |
| 사실과 다름  | 지적 내용이 실제 코드와 불일치                                     |
| 이미 해결됨  | 해당 문제가 이미 수정된 상태                                      |
| 대상 무관   | 검토 대상과 관련 없는 지적                                       |
| 과도한 추상화 | 간단한 인라인 코드를 함수로 분리, 과도한 인터페이스 도입 등                    |
| 새 기능 제안 | 사용자용 API/옵션 추가 같은 사용자 시점의 새로운 기능                      |
| 버그 생성   | 제안 내용이 버그인 지적                                         |
| 의미없는 제안 | 제안 내용이 "현재 상태 유지"를 의미하는 경우                            |

**문서화 처리 지침:**

- 의도된 동작이지만 사용자가 오해할 수 있는 경우 → 제외하지 말고 **"문서화 제안"**으로 유지
- 문서화 위치: 코드 관련 → TSDoc (우선) / 구조적·설계 관련 → README.md
- 코드관련 주석중 사용자가 알아야 할것은 반드시 TSDoc에 작성되어야함. 사용자는 몰라도 되는 내용은 일반 주석으로 작성

### 4. 결과 출력

재검토 결과를 바탕으로 중요도순으로 출력:

```markdown
## {{대상}} 검토 결과

### 제외 사항

| 제목 | 설명 | 제외이유 |
|------|------|----------|

### 문제점/개선점

**1. 문제 제목 (`위치`)**

- **문제**: {{현재 상태 설명}}
- **제안**: {{구체적 수정 방안}}
```

- `{{대상}}`: 프로젝트 혹은 패키지명
- `{{위치}}`: `파일명:라인번호`, `파일명:시작-끝`, `패키지명`, `여러 파일` 등 상황에 맞게 표기
- `{{구체적 수정 방안}}`: A또는 B라는 식의 여러가지 제안을 하는경우, 무엇이 권장되는지 반드시 표기

**주의:** 사용자의 피드백 없이 임의로 수정진행 절대 금지
