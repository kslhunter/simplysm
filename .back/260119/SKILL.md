# 검토 스킬

프로젝트/패키지의 문제점 및 개선점을 종합 검토하는 Task agent 관리 오케스트레이터이다.

- 이 오케스트레이터는 소스코드를 직접 분석하지 않는다.

## 작업 흐름

### 1. 범위 결정

- 범위가 명시되지 않으면 전체 프로젝트를 대상으로 한다.
- 패키지명이 명시되면 해당 패키지를 대상으로 한다.

### 2. 관점별 병렬 검토

Task(general-purpose)를 사용하여 4개 관점에 대해 **병렬로 실행**한다.

#### 2-1. 코드 검토

```
**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

{{대상}}에서 코드 검토:
- 에러 처리, 엣지 케이스 누락, 타입 안전성 등 코드 품질 전반
- 직접 구현 대신 표준 기능 또는 외부 npm 라이브러리로 대체 가능 여부
- simplysm 패키지 기능 우선 사용 권장 (필요시 의존성 추가)
- 코딩 지침(TYPESCRIPT_RULES.md) 준수 여부
- 프로젝트 내 유사 구현이 있으면 재사용 권장
- deprecated API 사용 여부
- 성능 최적화 가능 여부
```

#### 2-2. 아키텍처 검토

```
**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

{{대상}}에서 아키텍처 검토:
- 패키지 간 의존성 구조 적절성
- 순환 의존성 여부
- 계층 구조 및 책임 분리
- 확장성 및 유지보수성
```

#### 2-3. 테스트 검토

```
**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

{{대상}}에서 테스트 검토:
- 기존 테스트가 실제 기능을 충분히 커버하는지 확인
- 누락된 엣지 케이스나 시나리오에 대한 테스트 추가 필요 여부
- 단위테스트와 통합테스트 모두 확인
```

#### 2-4. 문서화 검토

```
**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

{{대상}}에서 문서화 검토:
- 공개 API에 대한 TSDoc 존재 여부
- README.md 내용의 정확성 및 최신성
- 복잡한 로직에 대한 주석 적절성
- 문서화 위치: 코드 관련 → TSDoc (우선) / 구조적·설계 관련 → README.md
- 코드관련 주석중 사용자가 알아야 할것은 반드시 TSDoc에 작성되어야함. 사용자는 몰라도 되는 내용은 일반 주석으로 작성 
```

### 3. 결과 취합 및 재검토

#### 취합

- 2단계 결과를 하나로 병합한다.
- 동일 위치에 대한 중복 지적을 제거한다.
- 유사한 제안을 통합한다.

#### 지적별 병렬 재검토

각 지적을 파일별로 묶어 Task(general-purpose)를 사용하여 **병렬로 재검토**한다.

```
**코딩지침:** `.claude/rules/TYPESCRIPT_RULES.md`

**제외 기준:**

| 기준      | 설명                                                    |
|---------|-------------------------------------------------------|
| 의도된 동작*  | TSDoc, 주석, README.md, 테스트, 명명 규칙 등으로 **명확한 증거** 확인 가능 |
| 사실과 다름  | 지적 내용이 실제 코드와 불일치                                     |
| 이미 해결됨  | 해당 문제가 이미 수정된 상태                                      |
| 대상 무관   | 검토 대상과 관련 없는 지적                                       |
| 과도한 추상화 | 간단한(3줄 이하) 인라인 코드를 함수로 분리, 과도한 인터페이스 도입 등                   |
| 새 기능 제안 | 사용자용 API/옵션 추가 같은 사용자 시점의 새로운 기능                      |
| 버그 생성   | 제안 내용이 버그인 지적                                         |
| 의미없는 제안 | 제안 내용이 "현재 상태 유지"를 의미하는 경우                            |
| 불가능한 가정 | 불가능한 상황에 대한 가정을 하는 경우 (예: 이미 타입에러로 입력된 경우에 대한 런타임 오류 수정) |

- 의도된 동작이지만 사용자가 오해할 수 있는 문제인 경우에는 제외하기보단 다음 분류에 따라 **문서화를 제안**하는 것으로 변경한다.
  - 코드관련 내용중 사용자가 알아야 할 것은 반드시 TSDoc으로 작성한다.
  - 코드관련 내용중 내부 로직등 사용자와 관련 없는 것들은 일반 주석으로 작성한다.
  - 그외 구조/설계 관련 내용은 README.md에 작성한다.

**재검토 절차:**

1. 해당 코드가 **왜 그렇게 작성되었는지**를 먼저 파악한다.
2. 기준에 따른 근거가 명확하다면 제외/수정한다.

- 여러가지 방법이 제안된 경우, 더 적당한 제안이 무엇인지 도출하여 한가지만 제안한다.

**출력 형식:**
제외사유: (제외된 경우만) 제외 기준 중 해당 항목
지적내용: 원본 또는 수정된 지적내용

```

### 4. 결과 출력

재검토 결과를 취합하여 분류별로 중요도순으로 정렬하여 출력한다.

**출력 양식:**

```markdown
# {{대상}} 검토 결과

## 제외 사항

| 제목 | 설명 | 제외이유 |
|------|------|----------|

## 문제점/개선점

### 코드/아키텍처

**1. 문제 제목 (`위치`)**

- **문제**: {{현재 상태 설명}}
- **제안**: {{구체적 수정 방안}}

### 테스트/문서화

{{개선내용}}
```

- 문제점/개선점의 각 항목의 번호는 관점간 공유한다. (1부터 다시 시작하지 않는다.)
- `{{대상}}`: 프로젝트 혹은 패키지명
- `{{위치}}`: `파일명:라인번호`, `파일명:시작-끝`, `패키지명`, `여러 파일` 등 상황에 맞게 표기

**주의:** 사용자의 피드백 없이 임의로 수정을 **절대** 진행하지 않는다.
