# 검토 스킬

프로젝트/패키지의 문제점 및 개선점을 종합 검토하는 오케스트레이터이다.

- 이 오케스트레이터는 소스코드를 직접 분석하지 않는다.
- 전문 agent들에게 검토를 위임하고 결과를 취합한다.

## 작업 흐름

### 1. 범위 결정

- 범위가 명시되지 않으면 전체 프로젝트를 대상으로 한다.
- 패키지명이 명시되면 해당 패키지를 대상으로 한다.

### 2. 관점별 병렬 검토

다음 4개 agent에게 **병렬로** 검토를 위임한다:

| Agent | 역할 | 위임 메시지 |
|-------|------|-------------|
| code-reviewer | 코드 검토 | `{{대상}}의 코드 품질을 검토해줘` |
| architect | 아키텍처 검토 | `{{대상}}의 아키텍처를 검토해줘` |
| qa-engineer | 테스트 검토 | `{{대상}}의 테스트를 검토해줘` |
| technical-writer | 문서화 검토 | `{{대상}}의 문서화를 검토해줘` |

### 3. 결과 취합

- 2단계 결과를 하나로 병합한다.
- 동일 위치에 대한 중복 지적을 제거한다.
- 유사한 제안을 통합한다.

### 4. 지적별 병렬 재검토

각 지적을 파일별로 묶어 `sd-review-validator` agent에게 **병렬로** 재검토를 위임한다.

```
Task(sd-review-validator):
파일: {{파일경로}}
지적 목록:
1. {{지적내용1}}
2. {{지적내용2}}
```

### 5. 결과 출력

재검토 결과를 취합하여 분류별로 중요도순으로 정렬하여 출력한다.

**출력 양식:**

```markdown
# {{대상}} 검토 결과

## 제외 사항

| 제목 | 설명 | 제외이유 |
|------|------|----------|

## 문제점/개선점

### 코드/아키텍처

**1. 문제 제목 (`위치`)**

- **문제**: {{현재 상태 설명}}
- **제안**: {{구체적 수정 방안}}

### 테스트/문서화

{{개선내용}}
```

- 문제점/개선점의 각 항목의 번호는 관점 간 공유한다. (1부터 다시 시작하지 않는다.)
- `{{대상}}`: 프로젝트 혹은 패키지명
- `{{위치}}`: `파일명:라인번호`, `파일명:시작-끝`, `패키지명`, `여러 파일` 등 상황에 맞게 표기

**주의:** 사용자의 피드백 없이 임의로 수정을 **절대** 진행하지 않는다.
