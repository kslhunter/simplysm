# /sd:review - 코드 심층 리뷰

$ARGUMENTS

코드를 면밀히 검토하여 문제점과 개선점을 찾는다.

## 트리거

- 코드 품질 검토 요청
- PR 전 리뷰
- 리팩토링 전 문제점 파악

## 인수 파싱

| 옵션 | 설명 |
|------|------|
| `--bug` | 잠재적 버그 관점 |
| `--logic` | 로직 이상 관점 |
| `--design` | 설계 문제 관점 |
| `--perf` | 성능 최적화 관점 |
| `--style` | 코드 일관성/규칙 준수 관점 |
| `--maintain` | 유지보수성 관점 |
| `--structure` | 구조적 최적화 관점 |
| `--test` | 테스트 필요성 및 테스트 상태 관점 |
| `--cleanup` | 불필요한 파일 삭제 검토 관점 |
| `--docs` | 문서 적절성 검토 관점 (CLAUDE.md, README.md) |
| `{패키지명}` | 대상 패키지 (와일드카드 지원: `service-*`) |

**관점 옵션 생략 시**: 전체 관점 적용

**출력 파일**: `REVIEW_{대상}_{YYYYMMDD}.md` (항상 저장)

## 분석 모드

| 모드 | 조건 | 분석 대상 |
|------|------|-----------|
| **전체 분석** | 패키지 미지정 | 아키텍처, 패키지 간 관계, 전역 일관성, 의존성 구조 |
| **패키지 분석** | 패키지 지정 | 해당 패키지 내부 코드 세부 분석 |

## 분석 관점

> 📖 상세 체크리스트: [review-checklist.md](../../references/review-checklist.md)

| 관점 | 옵션 | 전체 분석 시 | 패키지 분석 시 |
|------|------|-------------|---------------|
| 잠재적 버그 | `--bug` | 패키지 간 타입 불일치 | 엣지케이스, 리소스 누수 |
| 로직 이상 | `--logic` | 순환 의존성, 데드코드 | 조건문 오류, 경계값 |
| 설계 문제 | `--design` | 레이어 위반, 아키텍처 규칙 | SRP 위반, 결합도 |
| 성능 최적화 | `--perf` | 불필요한 의존성, 번들 크기 | 반복/재계산, N+1 |
| 코드 일관성 | `--style` | 패키지 간 네이밍/패턴 통일 | CLAUDE.md 규칙 준수 |
| 유지보수성 | `--maintain` | 공통화 가능 코드, 중복 | 복잡도, 매직 넘버 |
| 구조적 최적화 | `--structure` | 패키지 분리/통합, **책임 재배치** | 모듈 분리, API 설계 |
| 테스트 필요성 | `--test` | 테스트 커버리지 격차 | 테스트 제안, skip/todo |
| 불필요한 파일 | `--cleanup` | 전체 PLAN, MIGRATION 파일 | 패키지 내 임시 파일 |
| 문서 적절성 | `--docs` | 전체 문서 일관성 | 패키지 문서 최신성 |

## 실행 절차

### 1. 분석 모드 결정

**패키지명 미지정 → 전체 분석 모드:**
- 대상: 전체 코드베이스
- 출력: `REVIEW_codebase_{YYYYMMDD}.md`

**패키지명 지정 → 패키지 분석 모드:**
- 대상: `packages/{패키지명}/`
- 와일드카드: 패턴 매칭
- 출력: `REVIEW_{패키지명}_{YYYYMMDD}.md`

**제외 목록**: `.idea/*.iml`의 `excludeFolder` + `node_modules/`

### 2. 컨텍스트 파악

**전체 분석 모드:**
- 루트 `CLAUDE.md` 읽기 (아키텍처 규칙 파악)
- 각 패키지 `package.json` 분석 → 의존성 그래프 구성

**패키지 분석 모드:**
- 루트 `CLAUDE.md` + 패키지 `CLAUDE.md` 읽기

### 3. 코드 분석

`code-analyzer` 에이전트에게 분석 요청:
- 대상: 분석 모드에 따른 경로
- 관점: 선택된 옵션들
- 결과를 수집하여 다음 단계 진행

**전체 분석 모드 요청 내용:**
- 의존성 구조, 순환 의존성
- 레이어 위반 (CLAUDE.md 아키텍처 규칙 기준)
- 전역 패턴 일관성
- 공통화 가능 코드
- 로직 이동 후보

**패키지 분석 모드 요청 내용:**
- 코드 품질, 복잡도
- 테스트 상태
- 선택된 관점별 세부 분석

### 4. 결과 수집

선택된 관점에 따라 문제점/개선점 수집

### 5. 자체 검토 (중요)

**수집된 각 항목에 대해 검토**:

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| **타당성** | 실제 문제인가? 오탐 아닌가? |
| **맥락 적합성** | 이 코드베이스에서 의미 있는 지적인가? |
| **실현 가능성** | 수정 시 부작용 없는가? |
| **비용 대비 효과** | 수정 노력 대비 가치가 있는가? |

- 부적절한 항목 제외
- 애매한 항목은 "검토 필요" 섹션으로 분류
- `sequential-thinking` 활용

### 6. 결과 보고

**전체 분석 모드 출력 형식:**

```markdown
# Review: Codebase

## 요약
| 관점 | 발견 | 검토 후 확정 |
|------|------|-------------|
| 설계 문제 | 5건 | 3건 |

## 아키텍처 이슈

### #1. {제목}
**문제**: 설명
**위치**: `packages/{패키지}/src/{파일}.ts:{라인}`
**규칙**: CLAUDE.md 아키텍처 규칙 인용
**영향도**: 영향 범위

## 전역 일관성 이슈

### #2. {제목}
**문제**: 설명
**위치**: 여러 패키지
**제안**: 개선 방향

## 공통화 기회

### #3. {제목}
**문제**: 설명
**위치**: 해당 파일들
**제안**: 통합 방향

## 로직 이동 제안

### #4. {제목}
**현재 위치**: `packages/{상위}/src/{파일}.ts`
**이동 제안**: `packages/{하위}/src/{경로}`
**이유**: 순수 로직으로 환경 의존성 없음 / 여러 패키지에서 재사용 가능
**영향도**: 영향받는 패키지 목록

## 패키지 구조 이슈

### #5. {제목}
**문제**: 설명
**현재**: 현재 상태
**제안**: 개선 방향

## 제외됨 (참고)
| # | 원래 발견 | 제외 이유 |
|---|----------|----------|
| #6 | {내용} | {사유} |
```

**패키지 분석 모드 출력 형식:**

```markdown
# Review: {패키지명}

## 요약
| 관점 | 발견 | 검토 후 확정 |
|------|------|-------------|
| 잠재적 버그 | 5건 | 3건 |

## 반드시 확인 필요

### #1. {제목} (`path/file.ts:123`)
**문제**: 설명
**타당성**: 왜 실제 문제인지
**코드**:
```typescript
문제 코드
```
**제안**:
```typescript
개선 코드
```
**영향도**: 수정 시 영향 범위

## 개선 권장
...

## 선택적 개선
...
```

## 예외 처리

| 상황 | 대응 |
|------|------|
| 패키지 없음 | 사용 가능한 패키지 목록 표시 |
| 분석 파일 과다 | Task 에이전트로 병렬 분석 |
| 기존 REVIEW 파일 존재 | 덮어쓸지 확인 |
| 와일드카드 매칭 없음 | 사용 가능한 패턴 안내 |

## 사용 예시

```bash
# 전체 코드베이스 분석 (아키텍처, 패키지 간 관계)
/sd:review
/sd:review --design --structure

# 특정 패키지 세부 분석
/sd:review {패키지명}
/sd:review --bug {패키지 패턴}*
/sd:review --test {패키지명}
/sd:review --cleanup
/sd:review --docs {패키지명}
```
