# /sd:improve - 자동 개선 사이클

$ARGUMENTS

Review로 문제점 파악 → Plan 수립 → Task 실행 → Review 검증까지 자동 반복.

## 트리거

- 코드 품질 자동 개선
- 리팩토링 자동화

## 인수 파싱

| 옵션 | 설명 |
|------|------|
| `--bug` | 잠재적 버그 관점 |
| `--logic` | 로직 이상 관점 |
| `--design` | 설계 문제 관점 |
| `--perf` | 성능 최적화 관점 |
| `--style` | 코드 일관성/규칙 준수 관점 |
| `--maintain` | 유지보수성 관점 |
| `--structure` | 구조적 최적화 관점 |
| `--test` | 테스트 필요성 관점 |
| `--cleanup` | 불필요한 파일 삭제 관점 |
| `--docs` | 문서 적절성 관점 |
| `--max-cycles N` | 최대 사이클 수 (기본 5) |
| `--dry-run` | Review만 수행, 실제 수정 없음 |
| `{패키지명}` | 대상 패키지 (와일드카드 지원) |

**관점 옵션 생략 시**: 전체 관점 적용

## 분석 모드

| 모드 | 조건 | 분석 대상 | 수정 순서 |
|------|------|-----------|-----------|
| **전체 개선** | 패키지 미지정 | 아키텍처, 패키지 간 관계 | 의존성 분석 후 하위→상위 순서 |
| **패키지 개선** | 패키지 지정 | 해당 패키지 세부 | 해당 패키지만 |

## 실행 흐름

```
┌─────────────────────────────────────┐
│         Step 1: Review              │
│   개선점 0건 → 즉시 종료             │
│   개선점 N건 ↓                       │
├─────────────────────────────────────┤
│         Step 2: Plan                │
│   Review 결과 → 작업 계획 변환        │
├─────────────────────────────────────┤
│         Step 3: Task                │
│   전 Phase 자동 실행                 │
│              ↓                      │
│         Step 4: Review              │
│   N건 → Step 2로 (사이클 반복)       │
│   0건 ↓                             │
├─────────────────────────────────────┤
│         Step 5: Check               │
│   타입체크 → ESLint → 테스트          │
│   실패 → 자동 수정 시도               │
├─────────────────────────────────────┤
│         Step 6: Cleanup             │
│   PLAN, REVIEW 삭제 → 결과 보고      │
└─────────────────────────────────────┘
```

## 실행 절차

### Step 0: 분석 모드 결정

**패키지명 미지정 → 전체 개선 모드:**
- 대상: 전체 코드베이스
- 출력: `REVIEW_codebase_{YYYYMMDD}.md`, `PLAN_improve_codebase.md`
- 수정 순서: 각 패키지 `package.json` 분석 → 의존성 그래프 구성 → 하위 패키지부터 수정

**패키지명 지정 → 패키지 개선 모드:**
- 대상: `packages/{패키지명}/`
- 출력: `REVIEW_{패키지명}_{YYYYMMDD}.md`, `PLAN_improve_{패키지명}.md`

**제외 목록**: `.idea/*.iml`의 `excludeFolder` + `node_modules/`

### Step 1: Initial Review

`/sd:review` 로직 실행 (사용자 확인 생략):

**전체 개선 모드:**
- 아키텍처, 패키지 간 관계, 전역 일관성 분석
- `REVIEW_codebase_{YYYYMMDD}.md` 저장

**패키지 개선 모드:**
- 해당 패키지 세부 코드 분석
- `REVIEW_{패키지명}_{YYYYMMDD}.md` 저장

**분기:**
- 개선점 0건 → **즉시 종료** ("개선점 없음" 보고)
- 개선점 N건 → Step 2 진행
- `--dry-run` → 결과 보고 후 종료

### Step 2: Plan 생성

Review 결과를 Plan으로 변환:

**변환 규칙:**

| Review 섹션 | Plan Phase | 비고 |
|-------------|------------|------|
| 아키텍처 이슈 / 반드시 확인 필요 | Phase 1 | 필수 수정 |
| 전역 일관성 / 로직 이동 / 개선 권장 | Phase 2 | 구조 개선 |
| 공통화 기회 / 선택적 개선 | Phase 3 | 간단한 것만 |

**전체 개선 모드 Plan 형식:**

```markdown
# Plan: Improve Codebase (Cycle {N})

## 분석 결과
REVIEW_codebase_{YYYYMMDD}.md 기반 자동 생성

## 수정 순서
의존성 분석 결과 하위→상위 순서로 수정

## 구현 단계

### Phase 1: 아키텍처 수정
- [ ] #1 {제목} (`{패키지}/src/{파일}.ts`)
  - 수정 대상 패키지: {패키지}
- [ ] #2 {제목} (`{패키지}`, `{패키지}`)
  - 수정 순서: {하위 패키지} → {상위 패키지}

### Phase 2: 일관성/구조 개선
- [ ] #3 {제목} (여러 패키지)
  - 수정 순서: {하위} → {상위}
- [ ] #4 로직 이동: {함수명} (`{상위패키지}` → `{하위패키지}`)
  - 수정 순서: {하위} 추가 → {상위} 수정 → import 정리

### Phase 3: 공통화
- [ ] #5 {제목} (`{패키지}`, `{패키지}`)
  - 수정 순서: {하위} → {상위}

**검증**: `/sd:check`
```

- `PLAN_improve_{대상}.md` 저장 (덮어쓰기)
- TodoWrite에 작업 항목 등록

### Step 3: Task 실행

`/sd:task` 로직 실행 (사용자 확인 생략):

**전체 개선 모드:**
- 의존성 분석 결과에 따라 하위 패키지부터 수정
- 하위 패키지 수정 완료 후 상위 패키지 진행

**패키지 개선 모드:**
- 해당 패키지만 수정

- 모든 Phase 자동 진행 (Phase 간 확인 없음)
- 각 작업 완료 시 Plan 파일 체크박스 업데이트

**실패 처리:**

| 상황 | 대응 |
|------|------|
| 작업 실패 | 자체 분석 → 수정 시도 |
| 3회 연속 실패 | 해당 항목 스킵, 사유 기록 |
| 검증 실패 | 원인 분석 → 수정 → 재검증 (3회까지) |

→ 전체 완료 시 **Step 4로**

### Step 4: Validation Review

다시 Review 수행:

- 이전에 스킵된 항목은 제외하고 검토
- 동일 이슈 반복 감지 시 해결 불가 판단

**분기:**
- 개선점 N건 → **Step 2로** (다음 사이클)
- 개선점 0건 → **Step 5로**
- 최대 사이클 초과 → **Step 5로** (강제 종료)
- 동일 이슈만 반복 → **Step 5로** (해결 불가)

### Step 5: Final Check

`/sd:check` 실행:

**전체 개선 모드:**
- 전체 패키지 대상 타입 체크 → ESLint → 테스트

**패키지 개선 모드:**
- 해당 패키지 대상

- `/sd:check`의 실패 자동 수정 로직 활용

**실패 처리:**

| 상황 | 대응 |
|------|------|
| 타입 에러 | 자동 수정 시도 → 실패 시 보고 |
| ESLint 에러 | `--fix` 적용 → 실패 시 보고 |
| 테스트 실패 | 원인 분석 → 수정 시도 → 실패 시 보고 |

→ Check 완료 시 **Step 6으로**

### Step 6: Cleanup & Report

**파일 정리:**
- `PLAN_improve_{대상}.md` 삭제
- `REVIEW_{대상}_{YYYYMMDD}.md` 삭제

**결과 보고:**

**전체 개선 모드:**

```markdown
## Improve 완료: Codebase

### 사이클 요약
| Cycle | 발견 | 완료 | 스킵 |
|-------|------|------|------|
| 1     | 5    | 4    | 1    |
| 2     | 2    | 2    | 0    |
| 3     | 0    | -    | -    |

**총 {N}회 사이클**, 총 {M}건 개선

### 패키지별 수정 내역
| 패키지 | 수정 건수 | 주요 변경 |
|--------|----------|-----------|
| {패키지1} | 3 | {변경 요약} |
| {패키지2} | 1 | {변경 요약} |

### 스킵된 항목
| Cycle | 항목 | 사유 |
|-------|------|------|
| 1     | #3 {제목} | {사유} |

### 종료 사유
- [x] 개선점 0건 도달
- [ ] 최대 사이클 초과
- [ ] 해결 불가 판단
```

**패키지 개선 모드:**

```markdown
## Improve 완료: {패키지명}

### 사이클 요약
| Cycle | 발견 | 완료 | 스킵 |
|-------|------|------|------|
| 1     | 5    | 4    | 1    |
| 2     | 2    | 2    | 0    |
| 3     | 0    | -    | -    |

**총 {N}회 사이클**, 총 {M}건 개선

### 스킵된 항목
| Cycle | 항목 | 사유 |
|-------|------|------|
| 1     | #3 {제목} | {사유} |
```

## 스킵 항목 관리

**스킵 조건:**
1. 같은 작업 3회 연속 실패
2. 같은 이슈가 2사이클 연속 발견

**동일 이슈 판단:**
- 파일 경로 + 라인 범위(±10) 일치
- 또는 제목 유사도 80% 이상

스킵된 항목은 다음 Review에서 제외.

## 예외 처리

| 상황 | 대응 |
|------|------|
| 첫 Review 0건 | "개선점 없음" 보고 후 즉시 종료 |
| Task 3회 연속 실패 | 해당 항목 스킵, 다음 진행 |
| Plan 생성 불가 | 에러 보고 후 종료 |
| 5 사이클 초과 | 강제 종료, 현재까지 결과 보고 |
| 동일 이슈만 반복 | 해결 불가 판단, 스킵 후 종료 |

## 주의사항

- **완전 자동**: 모든 단계에서 사용자 확인 생략
- **반복 실행**: 개선점 0건까지 사이클 반복 (최대 N회)
- **자체 오류 해결**: 가능한 한 스스로 수정 시도
- **스킵 기록**: 해결 못한 항목은 최종 보고에 명시
- **파일 자동 정리**: 완료 시 PLAN, REVIEW 삭제
- **의존성 순서**: 전체 개선 시 `package.json` 분석 후 하위 패키지부터 수정

## 사용 예시

```bash
# 전체 코드베이스 개선 (아키텍처, 패키지 간 관계)
/sd:improve
/sd:improve --design --structure
/sd:improve --max-cycles 3

# 특정 패키지 세부 개선
/sd:improve {패키지명}
/sd:improve --bug {패키지 패턴}*
/sd:improve --dry-run {패키지명}
```
