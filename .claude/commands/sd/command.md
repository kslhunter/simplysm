# /sd:command - 커맨드 생성/개선

$ARGUMENTS

## 트리거

- 새 커맨드 생성 요청
- 기존 커맨드 개선/수정 요청

## 인수 파싱

| 인수 | 설명 |
|------|------|
| `--seq` | sequential-thinking 검토 강제 실행 |
| `{name}` | 커맨드명 언급 시 해당 커맨드 개선 |
| 자연어 | 기능 설명 시 새 커맨드 생성 |

## 실행 절차

### 1. 요청 분석

- 기존 커맨드명 언급 → `.claude/commands/sd/{name}.md` 읽기
  - 파일 없으면 → 새로 만들지 사용자에게 확인
- 새로운 기능 요청 → 초안 작성

### 2. 커맨드 작성/수정

아래 작성 가이드 기준으로:
- 기존 커맨드: 개선점 분석 후 수정안 작성
- 새 커맨드: 가이드에 맞춰 초안 작성

### 3. 검토

| 검토 항목 | 확인 내용 |
|-----------|-----------|
| frontmatter | `description` 필수 |
| 실행 절차 | 모호한 표현 없는지, 단계별 명확한 지시 |
| 예외 처리 | 에러 케이스 명시 |
| 사용 예시 | 실제 동작과 일치 |
| 작성 원칙 | 간결, 명령형, 병렬 표기 |
| 분리 검토 | reference 분리, 에이전트 분리 필요 여부 |

**`--seq` 옵션 시**: `sequential-thinking`으로 각 항목 단계별 검토

### 4. 사용자 확인 및 저장

- 검토 결과와 최종안 제시
- 승인 → `.claude/commands/sd/{name}.md`에 저장
- 에이전트 분리 시 → `.claude/agents/{name}.md`도 함께 저장
- 이름 충돌 시 → 덮어쓸지 확인

## 예외 처리

| 상황 | 대응 |
|------|------|
| 파일 없음 | 새로 만들지 확인 |
| 권한 오류 | 사용자에게 알림 |
| 문법 오류 | frontmatter 검증 후 수정 안내 |
| 에이전트 디렉토리 없음 | `.claude/agents/` 생성 후 진행 |

## 커맨드 작성 가이드

> 📖 공식 문서: [Custom Slash Commands](https://code.claude.com/docs/en/slash-commands#custom-slash-commands)

### Frontmatter

```yaml
---
description: 간단한 설명 (/help에 표시)
argument-hint: [--옵션] <자연어 요청>
---
```

| 필드 | 필수 | 설명 |
|------|------|------|
| `description` | ✅ | /help 목록에 표시 |
| `argument-hint` | 선택 | 자동완성 힌트 |

### 인수 패턴

```
/sd:name [--옵션들] <자연어 요청>
```

| 변수 | 설명 |
|------|------|
| `$ARGUMENTS` | 전체 인수 (문자열) |

**파싱 방법**: `$ARGUMENTS`에서 `--`로 시작하는 플래그 추출 후, 나머지는 자연어 요청으로 처리

### 본문 구조

    # /sd:name - 제목

    $ARGUMENTS

    ## 트리거
    - 언제 사용하는지

    ## 인수 파싱
    | 옵션 | 설명 |
    |------|------|
    | `--opt1` | 옵션1 설명 |

    ## 실행 절차
    ### 1. 단계명
    - 구체적 지시

    ## 예외 처리
    | 상황 | 대응 |
    |------|------|
    | {상황} | {대응} |

    ## 사용 예시
    /sd:name 자연어 요청

### 작성 원칙

- **간결하게**: Claude는 이미 똑똑함. 필요한 정보만
- **명령형**: "~합니다" 대신 "~실행", "~확인"
- **병렬 표기**: 독립 작업은 "(병렬 실행)"
- **네임스페이스**: `sd/` 폴더 → `/sd:command`

## Reference 분리 가이드

**목적**: 토큰 절약 - 필요할 때만 참조 파일 Read

**분리 판단 기준**:

| 기준 | 분리 | 설명 |
|------|:----:|------|
| **조건부 참조** | ✅ | 특정 옵션/상황에서만 필요 (예: `--bug` 시 버그 체크리스트) |
| **확장 가능성** | ✅ | 더 상세한 내용 추가로 품질 향상 가능 |
| **공유 가능성** | ✅ | 여러 커맨드에서 재사용 |
| **크기 초과** | ✅ | 200줄 이상 |
| **항상 필요** | ❌ | 매 실행마다 읽어야 하는 핵심 로직은 분리 안함 |

**디렉토리 구조**:

```
.claude/
├── commands/sd/{name}.md          # 메인 커맨드 (실행 흐름)
└── references/{name}-{section}.md # 분리된 참조 파일 (상세 내용)
```

**참조 방식** (커맨드 본문에서):

```markdown
> 📖 상세 내용: [참조 파일명](../../references/{name}-{section}.md)
```

## 에이전트 분리 가이드

**목적**: 전문 작업자 분리 - 재사용 가능한 역할 정의

### 커맨드 vs 에이전트

| 구분 | 커맨드 (작업 지시서) | 에이전트 (작업자) |
|------|---------------------|------------------|
| 역할 | "이 작업 해줘" | "나는 누구, 뭘 알고 있음" |
| 내용 | 절차, 옵션, 출력 형식 | 전문성, 규칙, 분석 방법 |
| 호출 | 사용자 `/command` | 커맨드에서 위임 |
| 재사용 | 단일 목적 | 여러 커맨드에서 사용 |

### 분리 판단 기준

| 기준 | 분리 | 설명 |
|------|:----:|------|
| **전문 역할** | ✅ | 특정 전문성이 필요한 작업 (코드 분석, 문서 작성 등) |
| **재사용성** | ✅ | 여러 커맨드에서 동일 역할 필요 |
| **컨텍스트 격리** | ✅ | 대량 출력이 메인 대화 오염 방지 필요 |
| **도구 제한** | ✅ | 특정 도구만 허용해야 할 때 (읽기 전용 등) |
| **단순 작업** | ❌ | bash 명령 몇 개로 끝나는 작업 |
| **절차 중심** | ❌ | 전문성보다 순서가 중요한 작업 |

### 에이전트 Frontmatter

```yaml
---
name: agent-name
description: 에이전트 설명 (Claude가 자동 위임 판단에 사용)
---
```

| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | ✅ | 유일한 식별자 (소문자, 하이픈) |
| `description` | ✅ | 역할 설명 (자동 위임 판단 기준) |

### 에이전트 본문 구조

```markdown
---
name: code-analyzer
description: 코드 품질 분석 전문가
---

나는 SIMPLYSM 프로젝트의 코드 분석 전문가입니다.

## 알고 있는 것
- CLAUDE.md의 코드 규칙
- 분석 관점별 체크리스트
- 아키텍처 레이어 규칙

## 분석 방법
- warpgrep으로 패턴 탐지
- 복잡도, 중첩 깊이 분석
- ...

## 출력 형식
- 문제점은 우선순위별 분류
- 코드 예시와 개선안 포함
```

### 커맨드에서 에이전트 호출

커맨드 본문에 자연어로 작성:

```markdown
### 3. 코드 분석

`code-analyzer` 에이전트에게 분석 요청:
- 대상: `packages/{패키지명}/`
- 관점: 선택된 옵션들
- 결과를 수집하여 보고서 작성
```

### 디렉토리 구조

```
.claude/
├── commands/sd/{name}.md  # 커맨드 (작업 지시)
├── agents/{name}.md       # 에이전트 (작업자)
└── references/            # 참조 파일 (상세 내용)
```

## 도구 선택 가이드

커맨드 작성 시 상황에 맞는 도구 지정:

| 상황 | 도구 |
|------|------|
| 전문 역할 위임 | 에이전트 (`.claude/agents/`) |
| 다단계 작업 추적 필요 | `TodoWrite` |
| 사용자 선택/확인 필요 | `AskUserQuestion` |
| 코드베이스 탐색/검색 | `warpgrep_codebase_search` (자연어 검색) |
| 파일 편집 | `morph edit_file` (빠른 편집, 10,500+ 토큰/초) |
| 복잡한 분석/의사결정 | `sequential-thinking` (MCP) |
| 외부 라이브러리 문서 조회 | `context7` (MCP) |

**morph 도구 상세**:

| 도구 | 용도 | 특징 |
|------|------|------|
| `warpgrep_codebase_search` | 자연어 기반 코드 검색 | "인증 처리하는 부분 찾아줘" 같은 질의 |
| `edit_file` | 파일 편집 | `// ... existing code ...` 플레이스홀더로 효율적 편집 |

파일/검색/명령 도구(Read, Write, Edit, Glob, Grep, Bash)는 별도 지정 없이 사용

## 사용 예시

```bash
/sd:command check 개선해줘
/sd:command --seq commit에 브랜치명 표시 추가해줘
/sd:command 빌드 후 배포하는 커맨드 만들어줘
/sd:command 코드 분석하는 에이전트 만들어줘
```
