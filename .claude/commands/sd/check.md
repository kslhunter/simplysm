# /sd:check - 패키지 품질 검증

$ARGUMENTS

> ⚠️ **핵심 원칙**: 오류 메시지만 보고 수정하지 않는다. 반드시 관련 코드를 읽고 원인을 검증한 후 수정한다.

## 트리거

- 품질 검증 요청
- 타입 체크, 린트, 테스트 실행

> 📖 오류 해결 가이드: [check-troubleshoot.md](../../references/check-troubleshoot.md)

## 인수 파싱

| 옵션 | 설명 |
|------|------|
| `--fix` | ESLint 자동 수정 |
| `--type` | 타입체크만 (조합 가능) |
| `--lint` | ESLint만 (조합 가능) |
| `--test` | 테스트만 (조합 가능) |
| `{name}` | 패키지/통합테스트명 (생략 시 전체) |

## 실행 절차

### 1. 대상 확인

- `packages/{name}` 존재 → 패키지
- `tests/{name}` 존재 → 통합테스트

### 2. 검증 (병렬 실행)

**ESLint 명령 선택** (프로젝트 판단: `package.json`의 `name` 필드):
| 프로젝트 | 명령 |
|----------|------|
| simplysm | `yarn run _sd-cli_ lint {경로}` |
| 그 외 | `sd-cli lint {경로}` |

> `--fix` 옵션 시: 명령 끝에 `--fix` 추가

**전체 검증**:
| 검증 | 명령 |
|------|------|
| 타입체크 | `npx tsc --noEmit` |
| ESLint | `{lint명령}` (경로 생략) |
| 테스트 | `npx vitest run` |

**패키지별** (`packages/{name}`):
| 검증 | 명령 |
|------|------|
| 타입체크 | `npx tsc --noEmit -p packages/{name}/tsconfig.build.json` |
| ESLint | `{lint명령} packages/{name}` |
| 테스트 | `npx vitest run packages/{name}` (tests/ 존재 시) |

**통합테스트** (`tests/{name}`):
- `npx vitest run tests/{name}`

> 하나가 실패해도 나머지는 계속 진행

### 3. 결과 보고

| 검증 | 결과 | 오류 수 |
|------|------|---------|
| 타입체크 | ✅/❌ | N |
| ESLint | ✅/❌ | N |
| 테스트 | ✅/❌ | N |

### 4. 오류 분석 (오류 발생 시)

#### 4.1 오류 분류 및 그룹화

- **같은 파일** 오류 그룹화
- **같은 오류 코드** (TS2307, TS2345 등) 그룹화
- 오류 수가 많으면 **패턴 분석**으로 근본 원인 추정

#### 4.2 근본 원인 식별

| 패턴 | 근본 원인 가능성 |
|------|-----------------|
| 같은 모듈 import 오류 다수 | 해당 모듈 문제 (export 누락, 경로) |
| 같은 타입 관련 오류 다수 | 타입 정의 변경/누락 |
| 한 파일 수정 후 다른 파일 오류 | 인터페이스 변경 → 파생 오류 |
| 단일 파일 단일 오류 | 해당 파일 내부 문제 |

**파생 오류 판단**: 근본 원인 1개 수정 시 해결될 오류들은 별도 수정 불필요

#### 4.3 가설 검증 (필수)

**절대 오류 메시지만 보고 수정하지 않는다.** 다음 절차 필수:

1. **오류 발생 코드 읽기** - `Read` 도구로 해당 파일/라인 확인
2. **관련 정의 확인** - import된 타입, 함수의 원본 정의 확인
3. **가설 수립** - "이것이 원인이다"
4. **가설 검증** - 코드가 가설을 지지하는지 확인

| 오류 유형 | 필수 확인 항목 |
|-----------|---------------|
| TS2307 (모듈 못 찾음) | 대상 모듈의 `index.ts` export 목록, `tsconfig.json` paths |
| TS2345 (타입 불일치) | 함수 시그니처, 전달 값의 실제 타입 |
| TS2339 (속성 없음) | 대상 타입의 interface/type 정의 |
| ESLint 오류 | 오류 발생 코드 주변 문맥 (5줄 이상) |
| 테스트 실패 | 테스트 코드 + 테스트 대상 함수 |

#### 4.4 확신도 표시

| 확신도 | 조건 | 대응 |
|--------|------|------|
| ✅ 높음 | 코드 확인 완료, 원인 명확 | 수정안 제시 후 승인 요청 |
| ⚠️ 중간 | 코드 확인했으나 복잡 | 수정안 + 대안 함께 제시 |
| ❓ 낮음 | 추측 단계 | 사용자에게 확인 먼저 요청 |

**확신도 낮음인 경우**: 수정을 시도하기 전에 반드시 사용자에게 가설 설명 후 확인

### 5. 수정 및 재검증

1. **승인 후 수정** - 사용자 승인 없이 수정하지 않음
2. **수정 후 재검증** - 동일 검증 명령 재실행 (필수)
3. **새 오류 발생 시** - 원인 분석 재시작 또는 롤백 고려

## 예외 처리

| 상황 | 대응 |
|------|------|
| 패키지 없음 | 사용 가능한 패키지 목록 표시 |
| 명령 실패 | 나머지 검증 계속 → 실패 원인 보고 |
| ESLint 자동 수정 불가 | 수동 수정 필요 항목 안내 |
| 테스트 타임아웃 | 해당 테스트 스킵 후 계속, 원인 보고 |
| 오류 원인 불명확 | 확신도 "낮음" 표시, 사용자 확인 요청 |

## 사용 예시

```bash
/sd:check                      # 전체 검증
/sd:check --fix                # ESLint 자동 수정 포함
/sd:check core-common          # 특정 패키지
/sd:check core-common --type   # 타입체크만
/sd:check --type --lint        # 타입체크 + ESLint
/sd:check orm                  # 통합테스트
```
