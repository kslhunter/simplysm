# Eval: sd-wbs

## 행동 Eval

### 시나리오 1: 상세한 요구사항으로 WBS 생성
- 입력: "/sd-wbs 사내 도서 관리 시스템을 만들려고 한다. 목표는 도서 대출 반납 처리 시간을 50% 단축하는 것이다. 사용자는 일반 직원(대출/반납)과 도서 관리자(도서 등록/재고 관리)이다. 직원은 모바일에서 바코드 스캔으로 대출/반납하고, 관리자는 웹에서 재고를 관리한다. 알림 기능, 통계 대시보드는 제외한다."
- 체크리스트:
  - [ ] `.tasks/` 하위에 `{yyMMddHHmmss}_{topic}/wbs.md` 경로로 파일을 생성한다
  - [ ] Impact Mapping에 Goal, Actor, Impact, Deliverable이 모두 포함된다
  - [ ] Goal이 측정 가능하게 기술된다 ("처리 시간 50% 단축" 수준)
  - [ ] Impact가 기능이 아닌 행동 변화로 기술된다
  - [ ] Feature Breakdown에 `[ ]` 체크박스가 있다
  - [ ] Feature 번호 순서가 구현 순서이다 (앞 Feature가 뒤 Feature의 기반)
  - [ ] 각 Feature가 이전 Feature만 완료된 상태에서 독립 구현 가능하다
  - [ ] 범위 힌트가 기능적 역할(What)만 기술하고 기술명/구현방법(How)을 포함하지 않는다
  - [ ] 제외 사항에 "알림 기능", "통계 대시보드"가 포함된다
  - [ ] 다음 단계로 `/sd-dev-spec` 또는 `/sd-dev`를 안내한다
  - [ ] wbs.md에 `## 참조 자료` 섹션이 포함된다
  - [ ] 입력에서 언급된 구체적 정보(플랫폼: 모바일/웹, 디바이스 조작: 바코드 스캔)가 범위 힌트가 아닌 참조 자료에 기록된다

### 시나리오 2: 수평 분해가 유혹되는 라이브러리 마이그레이션
- 사전 조건: `.tasks/260101120000_lib-migration/source-structure.md` 파일 존재 (유틸리티 5개 + 컴포넌트 10개로 구성된 UI 라이브러리, 컴포넌트들이 유틸리티를 사용)
- 입력: "/sd-wbs .tasks/260101120000_lib-migration/source-structure.md 파일을 참고하여 이 UI 라이브러리를 새 워크스페이스로 마이그레이션하는 WBS를 작성해줘. 목표는 v2 워크스페이스에서 모든 컴포넌트를 사용할 수 있게 하는 것이다."
- 체크리스트:
  - [ ] Feature가 기술 레이어(유틸리티 전부, 컴포넌트 전부)로 분류되지 않는다
  - [ ] 각 Feature가 자기 필요한 유틸리티를 포함하거나 공유 인프라 Feature 이후에 배치된다
  - [ ] Feature 번호 순서가 구현 순서이다
  - [ ] 각 Feature가 이전 Feature만 완료된 상태에서 독립 구현 가능하다
  - [ ] 15개 소스 모듈에 대해 Feature가 개별 모듈 단위로 분해되어 있다 (기능 범주 3~4개가 아닌, 밀접 결합 모듈 단위로 10개 이상)
  - [ ] 하나의 Feature에 독립적으로 동작하는 모듈 여러 개가 포함되지 않는다 (단, 수직 슬라이스 — 유틸리티 + 첫 소비 모듈 — 는 밀접 결합으로 간주)
  - [ ] 참조 파일 경로(`source-structure.md`)가 참조 자료에 기록된다
  - [ ] 참조 파일 경로에 이 파일에서 무엇을 확인해야 하는지(목적)가 함께 기록된다

### 시나리오 3: 순환 의존이 발생하는 요구사항
- 입력: "/sd-wbs 프로젝트 관리 시스템. 목표는 프로젝트 진행 가시성을 80% 향상. 사용자는 PM과 팀원. 기능: 사용자 관리(사용자 CRUD, 로그인 시 권한 검증), 권한 관리(사용자별·역할별 권한 규칙 설정), 프로젝트 CRUD, 작업 할당. 외부 연동은 제외."
- 체크리스트:
  - [ ] 사용자 관리 ↔ 권한 관리 사이의 순환 의존 가능성을 인식한다
  - [ ] 순환을 해소하는 방법(Feature 분할, 기능 재배치 등)을 적용하거나, 순환이 발생하지 않는 설계를 채택한다
  - [ ] 해소 후 Feature 번호 순서대로 구현 시 순환이 발생하지 않는다
  - [ ] 해소 방법과 그 근거를 출력에서 설명한다

### 시나리오 4: 구체적 정보가 풍부한 요구사항
- 입력: "/sd-wbs 물류 창고 관리 시스템. 기존 레거시 시스템(legacy-wms/ 폴더)을 마이그레이션한다. 목표는 입출고 오류율을 5% 이하로 줄이는 것이다. 사용자는 현장 작업자와 물류 관리자. 바코드는 Code128과 QR 두 종류를 사용한다. 중량 단위는 kg과 lb를 지원하고 입고 시 중량 오차 허용 범위는 ±2%이다. 외부 ERP(SAP)와 REST API로 연동한다. 현장 PDA는 Android 기반이다. 통관 업무는 제외."
- 체크리스트:
  - [ ] wbs.md에 `## 참조 자료` 섹션이 포함된다
  - [ ] 구체적 데이터(바코드 종류: Code128/QR, 중량 단위: kg/lb, 오차 범위: ±2%)가 참조 자료에 기록된다
  - [ ] 기술 제약(REST API, SAP 연동, Android PDA)이 참조 자료에 기록된다
  - [ ] 참조 파일 경로(`legacy-wms/`)가 목적과 함께 참조 자료에 기록된다
  - [ ] 범위 힌트에는 "바코드 스캔", "중량 관리" 등 기능적 역할만 기술되고, Code128/QR/kg/lb 등 구체 열거는 포함되지 않는다
  - [ ] Feature 번호 순서가 구현 순서이다
  - [ ] 제외 사항에 "통관 업무"가 포함된다
