# 종합 검토

이 스킬은 **오케스트레이터**로서 `/sd-improve-scan`과 `/sd-improve-validate` 스킬을 호출하여 검토를 수행한다.

## 인자

- `$ARGUMENTS` (선택): 검토 대상 패키지명 (예: `core-common`)
  - 지정 시: `packages/<패키지명>` 디렉토리를 대상으로 검토
  - 미지정 시: 프로젝트 전체를 대상으로 검토

## 제한 사항

- 직접 코드나 파일을 읽지 않는다.
- 직접 검토를 수행하지 않는다.
- 아래 "호출 양식"은 `/sd-improve-scan` 호출 시 인자로 전달할 내용이다.

## 작업 흐름

각 검토 호출 양식(logic, code, architect, test, docs)에 대해:

1. `/sd-improve-scan` 호출 (아래 호출 양식 참조) → 지적 목록 받음
   - 지적이 없을경우, 보고 후 종료.
2. `/sd-improve-validate`에 지적 목록 전달 → 검증된 목록 받음
   - 검증된 지적이 없을경우, 보고 후 종료.
3. 검증된 지적 각각에 대해:
   - 아래 형식으로 출력 후 수정 수행:
     ```
     ### 수정 #N: {{문제 제목}}
     - 위치: {{파일:라인}}
     - 문제: {{현재 상태}}
     - 수정: {{수정 내용}}
     ```
   - 출력 후 해당 수정 작업 수행
4. 1번부터 다시 반복

## logic 호출 양식

```
대상: {{대상 경로}}
참고자료:
- .claude/rules/TYPESCRIPT_RULES.md
지침:
- 테스트 코드는 검토 대상이 아님.
- 로직 정확성: 조건문/반복문 흐름이 의도대로 동작하는지, off-by-one 에러, 경계 조건 오류
- 알고리즘 개선: 동일한 결과를 더 효율적이거나 명확하게 달성할 수 있는 방법
- 로직 단순화: 불필요하게 복잡한 로직을 단순화할 수 있는지
- 의도 명확성: 코드가 작성자의 의도를 명확히 표현하는지
- 중복 로직: 10줄 이상의 유사한 로직이 2회 이상 반복되거나, 짧더라도 3회 이상 반복되는 경우 추상화 검토
- 조건 표현: 복잡한 조건문을 더 읽기 쉽게 재구성할 수 있는지
- API 설계: 함수/메서드의 입출력 타입이 사용자 입장에서 적절한지, 더 나은 인터페이스 제공 방법이 있는지
- 사용 편의성: 호출자가 불필요한 변환이나 준비 작업 없이 쉽게 사용할 수 있는지
```

## code 호출 양식

```
대상: {{대상 경로}}
참고자료:
- .claude/rules/TYPESCRIPT_RULES.md
- .claude/rules/SD_PKG_IDX.md
지침:
- 테스트 코드는 검토 대상이 아님.
- 에러 처리: try-catch 누락, 에러 전파 방식, 에러 메시지 품질
- 엣지 케이스: null/undefined 처리, 빈 배열/객체 처리, 경계값 처리
- 타입 안전성: any 사용, 타입 가드 누락, 타입 캐스팅 남용
- 표준 기능 대체: 직접 구현 대신 표준 기능 또는 외부 npm 라이브러리로 대체 가능 여부
- simplysm 패키지 활용: @simplysm/* 패키지 기능 우선 사용 권장 (필요시 의존성 추가)
- 지침 준수: TYPESCRIPT_RULES.md 준수 여부
- 코드 재사용: 프로젝트 내 유사 구현이 있으면 재사용 권장
- deprecated API: deprecated API 사용 여부
- 성능: 불필요한 연산, 메모리 누수 가능성, 비효율적 알고리즘
```

## test 호출 양식

```
대상: {{대상 경로}}
참고자료:
- .claude/rules/TYPESCRIPT_RULES.md
- CLAUDE.md
지침:
- 테스트 커버리지: 기존 테스트가 실제 기능을 충분히 커버하는지 확인
- 엣지 케이스: 누락된 엣지 케이스나 시나리오에 대한 테스트 추가 필요 여부
- 단위 테스트: 개별 함수/클래스에 대한 테스트 적절성
- 통합 테스트: 모듈 간 상호작용 테스트 적절성 (프로젝트 루트의 tests/ 검토 필요)
- 테스트 품질: 테스트 코드의 가독성, 유지보수성, 중복 여부
- 테스트가 문제인지 사용자 구현이 문제인지 면밀히 파악할 것.
```

## docs 호출 양식

```
대상: {{대상 경로}}
참고자료:
- .claude/rules/TYPESCRIPT_RULES.md
- .claude/rules/MD_RULES.md
지침:
- TSDoc: 공개 API에 대한 TSDoc 존재 여부 및 품질
- README: README.md 내용의 정확성 및 최신성
- 주석 적절성: 복잡한 로직에 대한 주석 적절성
- 문서화 위치: 코드 관련 내용 중 사용자가 알아야 할 것은 TSDoc (우선), 내부 로직은 일반 주석, 구조적/설계 관련 내용은 README.md
- 직역등 어색한 표현을 확인, 수정한다.
- 로직이 복잡한 경우, TSDoc과 주석을 적극 활용한다.
```
