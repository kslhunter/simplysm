# sd-spec: 요구분석서 작성

사용자 요청과 코드베이스를 분석하여 최대한 상세한 요구분석서(`spec.md`)를 작성한다.

## 워크플로우

### 1. 명확화 질문

**핵심 원칙: 추측 금지. 애매하면 반드시 질문한다.**

코드베이스 분석과 사용자 요청을 종합한 뒤, 아래 체크리스트를 반드시 검토하여 질문이 필요한 항목을 도출한다.
질문 없이 바로 문서 작성으로 넘어가는 것은 예외적 상황이다.

#### 필수 검토 항목

각 항목에 대해 "사용자가 명시적으로 언급했는가?"를 확인한다.
언급하지 않았고, 코드베이스에서도 유일한 답이 도출되지 않으면 질문한다:

| 검토 항목 | 질문이 필요한 경우 |
|-----------|-------------------|
| **범위(scope)** | 어디까지 포함하는지 경계가 불분명 |
| **대상 사용자/환경** | 누가, 어떤 환경에서 사용하는지 불명확 |
| **기존 동작과의 관계** | 대체/확장/병존 중 어떤 것인지 불명확 |
| **에러/예외 처리** | 실패 시 동작이 명시되지 않음 |
| **UI/UX 세부사항** | 화면 관련인데 레이아웃/흐름이 불명확 |
| **데이터 모델/스키마** | DB/모델 변경이 수반되는데 구조 불명확 |
| **성능/제약 조건** | 대량 데이터, 동시성 등 비기능 요구사항 불분명 |
| **의존성/호환성** | 다른 패키지/시스템 연동 방식 불명확 |

#### 질문 프로세스

**한번에 1개씩만** 다음 프로세스를 반복한다.

1. 질문 대상에 대한 **자세한 설명을 텍스트로 먼저 출력**한다
   - 선택지 각각의 장단점, 트레이드오프, 10점만점의 점수를 설명한다
   - 현재 코드베이스의 맥락을 반영한 설명을 제공한다
2. AskUserQuestion을 호출한다 (한번에 하나씩만 질문)
3. 답변을 반영하여 추가 분석을 수행한다
4. 불명확한 점이 더 있으면 1번으로 돌아간다

**모든 검토 항목을 확인했고, 사용자의 요청이 충분히 구체적일 때만** 질문을 종료하고 문서 작성으로 넘어간다.

### 2. 요구분석서 작성 및 저장

#### 참조 문서

타 문서를 기반으로 작성하는 경우, 문서 상단에 참조를 추가한다:

```markdown
**참조 문서:** `{경로}` {대상섹션번호}
```

#### 문서 템플릿

```markdown
# 요구분석서: {제목}

## 개요

기능의 목적과 배경을 기술한다.

## 현재 상태

코드베이스 분석 결과를 기술한다.
- 관련 기존 코드/패키지
- 현재 구현 방식
- 관련 의존성

## 요구사항

### R1. {요구사항 제목}

{상세 내용}

#### 구현 체크리스트
- [ ] {구현 항목 1}
- [ ] {구현 항목 2}

#### 영향 범위

| 경로 | 변경 내용 |
|------|----------|
| {파일/패키지 경로} | {변경 설명} |
```

#### 구현 체크리스트 규칙

각 R항목의 상세 내용을 기술한 후, 해당 R에서 구현해야 할 개별 항목을 `- [ ]` 형식의 체크리스트로 빠짐없이 나열한다.

- 고정 템플릿이 아니다. R항목의 상세 내용에서 구현 항목을 추출하여 작성한다
- 상세 내용에 기술된 모든 기능, 동작, 처리를 빠짐없이 개별 항목으로 분리한다
- 후속 작업(sd-plan, sd-plan-dev)이 이 체크리스트를 참조하여 누락 없이 구현한다

#### R번호 부여 기준

- **기본은 단일 R이다.** 나눌 명확한 이유가 없으면 R1 하나로 작성한다
- 동일 기능의 구현 단계(모델→로직→UI)를 별도 R로 나누지 않는다
- 서로 다른 기능/관심사일 때만 R을 나눈다

**잘못된 분리:**

```markdown
### R1. 설정 데이터 모델 정의
### R2. 설정 저장/불러오기 구현
### R3. 설정 UI 컴포넌트 작성
```

→ 하나의 구현 단위다. 합친다: `### R1. 설정 기능 구현`

**올바른 분리:**

```markdown
### R1. 사용자 인증 (회원가입, 로그인, 토큰 발급)
### R2. 역할 기반 접근 제어 (권한 정의, 미들웨어, 라우트 가드)
```

→ 각각 독립 구현·검증 가능, 합치면 범위 과도

#### 추가 섹션 (점진적 공개)

요구사항의 특성을 판단하여 해당하는 참조 파일만 Read하고, 그 양식에 따라 추가 섹션을 작성한다:

| 요구사항 특성 | 참조 파일 |
|-------------|----------|
| UI/화면 관련 | `${CLAUDE_SKILL_DIR}/sections/ui.md` |
| API/엔드포인트 관련 | `${CLAUDE_SKILL_DIR}/sections/api.md` |
| DB/테이블/스키마 관련 | `${CLAUDE_SKILL_DIR}/sections/db.md` |
| 프로세스/흐름 관련 | `${CLAUDE_SKILL_DIR}/sections/process.md` |
| 아키텍처/패키지 구조 관련 | `${CLAUDE_SKILL_DIR}/sections/architecture.md` |

#### 첨부파일

대화 중 등장한 파일(사용자 첨부, sd-doc-extract 추출 등)이 있으면:
1. 저장 디렉토리의 `attachments/` 폴더에 복사한다
2. 문서 내 관련 위치에 링크한다: `![{파일명}](attachments/{파일명})` 또는 `[{파일명}](attachments/{파일명})`

#### 저장 경로

topic은 영문 kebab-case로 변환, 타임스탬프는 `date +%y%m%d%H%M%S`로 생성한다.

```
신규:           .tasks/{yyMMddHHmmss}_{topic}/spec.md
R항목 상세화:    {spec 디렉토리}/{R번호}/spec.md
migration.md:   {migration.md 디렉토리}/[{M번호}/]spec.md  (복수 M이면 M1_M5/ 형태)
```

#### 완료 안내

문서 저장 후 다음을 출력한다:
- 파일 경로
- 다음 단계:
  ```
  /sd-plan {spec경로}              # 구현계획 수립
  /sd-plan {spec경로} R1           # 특정 R항목만 구현계획 수립
  /sd-spec {spec경로} R1           # 특정 R항목을 상세 spec으로 분리
  ```
