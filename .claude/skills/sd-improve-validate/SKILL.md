# 지적 검증

이 스킬은 지적을 검증한다. 반드시 이 문서를 꼼꼼히 모두 읽어보고 검증해야 한다.

## 작업 흐름

1. 지적당한 코드를 좀더 세밀히 분석한다. (대상 파일 전체 내용 / 관계 파일 내용 / README.md 문서 확인)
2. 제외근거, 수정지침을 통해 지적을 제외하거나 수정한다.
3. 검증 결과를 출력한다.
   - Plan mode와 관계없이 직접 검토를 수행하고 결과만 반환한다. plan 파일을 쓰지 않는다.

## 제외 근거

| 기준         | 설명                                                       |
|------------|----------------------------------------------------------|
| 의도된 동작     | TSDoc, 주석, README.md, 명명 규칙 등에 **명확한 의도**가 기술되어 있는 경우    |
| 사실과 다름     | 지적 내용이 실제 코드와 일치하지 않는 경우                                 |
| 이미 해결됨     | 해당 문제가 이미 해결된 상태인 경우                                     |
| 대상 무관      | 검토 대상과 관련 없는 지적                                          |
| 과도한 추상화    | 간단한 인라인 코드를 함수로 분리, 과도한 인터페이스 도입 등                       |
| 새 기능 제안    | 사용자용 API/옵션 추가. (**주의:** 안정성 향상은 새 기능으로 볼 수 없음)          |
| 버그 생성      | 제안 내용이 버그인 지적                                            |
| 의미없는 제안    | 제안 내용이 "현재 상태 유지"를 의미하는 경우                               |
| 불가능한 상황 가정 | 절대 발생할 수 없는 상황을 가정한 경우 (예: 이미 타입에러 상태인 경우에 대한 런타임 오류 수정) |
| 근거없는 문제인식  | 문제로 인식하였으나 그것이 문제라는 근거가 없거나 잘못된 경우                       |

## 문서화 제안 지침

문서화 제안인 경우 다음 지침을 따른다:

- 코드 관련 내용 중 사용자가 알아야 할 것: TSDoc으로 작성
- 코드 관련 내용 중 내부 로직: 일반 주석으로 작성
- 그 외 구조/설계 관련 내용: README.md에 작성

## 출력 형식

```markdown
### 제외된 지적

| 번호 | 지적내용 | 제외사유 |
|------|----------|----------|
| [관점] 1 | 원본 지적내용 | 제외 기준 중 해당 항목 |

### 유효한 지적

**[관점] 2. 문제 제목 (`위치`)**

- **문제**: 현재 상태 설명
- **근거**: {{판단 근거}}
- **제안**: 구체적 수정 방안
```
