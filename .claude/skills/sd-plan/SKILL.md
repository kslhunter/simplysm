# sd-plan: 구현 계획서 작성

이 스킬의 목적은 요구 명세서(spec.md)를 **LLM이 기계적으로 구현할 수 있는 수준의 구현 계획서(plan.md)**로 변환하는 것이다.

plan.md의 핵심은 세 가지이다:
- **Contracts**: 산출물의 구조를 먼저 정의한다. 이것이 LLM에게 주는 가장 명확한 구현 타겟이다.
- **Slices**: 독립적으로 구현 가능한 작업 단위로 분해한다. 이 단위가 병렬 Agent 실행의 기본 단위가 된다.
- **Verification List**: spec의 GWT 시나리오를 구체적 검증 항목으로 변환한다. `/sd-plan-dev`에서 이 목록을 기반으로 검증 우선(Verify-First) 방식으로 구현한다.

한국어로 응답한다.

## AskUserQuestion 사용 규칙

질문이 여러 개일 때 반드시 **한 번에 하나씩** 순차적으로 진행한다:
1. 하나의 질문에 대한 설명을 출력한다.
2. `---`를 출력한다.
3. 하나의 AskUserQuestion을 호출한다.
4. 답변을 받은 후, 그 답변을 반영하여 다음 질문으로 넘어간다.

앞선 답변이 이후 질문의 내용이나 필요성에 영향을 줄 수 있으므로, 여러 질문을 하나의 AskUserQuestion에 묶지 않는다.

## 전체 프로세스 맥락

이 스킬은 3단계 파이프라인의 두 번째이다:
1. `/sd-spec` → 요구 명세서(spec.md)
2. **`/sd-plan`** (이 스킬) → 구현 계획서(plan.md)
3. `/sd-plan-dev` → plan.md 기반 구현

spec.md의 Given/When/Then이 plan.md의 Verification List로 변환되고, Contracts가 산출물의 구조를 확정한다. `/sd-plan-dev`는 plan.md를 완벽한 것으로 간주하고 질문 없이 구현하므로, plan.md에 모호함이 있으면 안 된다.

## 입력

`$ARGUMENTS`로 spec.md 파일 경로를 받는다.

## 프로세스

### 1단계: leaf 검증

spec.md가 있는 디렉토리에서 `R*/spec.md` 패턴의 하위가 존재하는지 Glob으로 확인한다.

- **하위가 존재** → 이 spec은 더 작은 단위로 분해된 상태이므로, leaf spec 경로를 안내하고 중단한다.
- **하위 없음** → leaf. 다음 단계로 진행한다.

leaf만 받는 이유: 분해된 상위 spec은 하위 spec들의 합이므로, 상위를 직접 plan하면 중복이 생긴다.

### 2단계: 맥락 수집

1. spec.md를 읽는다.
2. `> Parent:` 링크가 있으면 상위 spec.md도 읽는다 (재귀적으로 최상위까지). 상위 spec은 전체 맥락을 제공하여 하위 spec의 의도를 더 잘 이해하게 해준다.
3. 프로젝트의 기존 파일을 탐색하여 재사용 가능한 모듈, 유틸리티, 패턴을 파악한다. 새로 만들 것이 적을수록 좋은 계획이다.

### 3단계: 기술적 명확화

spec에서 **기술적 관점**으로 불명확한 점이 있으면 AskUserQuestion으로 질문한다.

이 단계의 질문은 `/sd-spec`의 질문과 관점이 다르다:
- `/sd-spec`은 "무엇을 만들지" (비즈니스/사용자 관점)
- `/sd-plan`은 **"어떻게 만들지"** (기술/구현 관점)

기술적 관점 질문 예시:
- "이 기능에 사용할 라이브러리/프레임워크 선호가 있나요?"
- "기존 A 모듈을 재사용할까요, 새로 만들까요?"
- "성능/동시성 요구사항이 있나요?"
- "이 산출물의 형식(파일 포맷, 디렉토리 구조)에 대한 제약이 있나요?"
- "기존 코드/설정에 X 패턴이 있는데, 이를 따를까요?"

### 4단계: plan.md 작성

spec.md와 **같은 디렉토리**에 plan.md를 작성한다.
`${CLAUDE_SKILL_DIR}/template-plan.md`를 읽고 동일한 형식으로 작성한다.

### 5단계: spec ↔ plan 정합성 검증

plan.md 작성 후, spec.md와 대조하여 누락이나 초과가 없는지 검증한다. `/sd-plan-dev`는 plan.md를 완벽한 것으로 간주하므로, 여기서 잡지 못한 누락은 최종 산출물에서도 빠진다.

1. **GWT 누락 없음**: spec의 모든 Scenario(Given/When/Then)가 Verification List에 대응 항목이 있는지 확인한다. 빠진 것이 있으면 추가한다.
2. **Rule 누락 없음**: spec의 모든 Rule이 최소 하나의 Slice에서 다뤄지는지 확인한다. 빠진 Rule이 있으면 Slice를 추가하거나 기존 Slice에 포함한다.
3. **Constraint 반영**: spec의 모든 Constraint가 plan의 Contracts 또는 Verification List에 반영되었는지 확인한다.
4. **범위 초과 없음**: plan에 spec에 없는 기능이나 요구사항이 추가되지 않았는지 확인한다. 추가된 것이 있으면 제거한다.

## plan.md 각 섹션 작성 가이드

### Spec 섹션

원본 spec.md의 경로를 기록한다. `/sd-plan-dev`가 필요 시 원본 spec을 참조할 수 있도록 한다.

### Contracts 섹션

**왜 산출물 구조를 먼저 정의하는가**: LLM에게 명확한 구현 타겟을 주는 것이 이 섹션의 목적이다. "로그인 기능을 만들어라"보다 구체적인 구조 정의가 해석의 여지를 없앤다.

산출물의 형태에 맞게 구조를 정의한다:
- 코드라면: 인터페이스/타입 정의, 함수 시그니처, 입출력 타입, 에러 타입
- 스킬이라면: 파일 구조, frontmatter 필드, 본문 섹션 구조
- 기존 프로젝트의 코드/패턴을 최대한 재사용한다

### Slices 섹션

**왜 독립 작업 단위로 분해하는가**: `/sd-plan-dev`는 독립 Slice들을 병렬 Agent로 동시 실행한다. Slice가 진정으로 독립적이어야 충돌 없이 병렬 실행이 가능하다. 하나의 Slice는 "하나의 Agent가 다른 Slice를 모르는 상태에서도 구현할 수 있는 단위"여야 한다.

각 Slice에 명시할 필드:
- **Files**: 생성하거나 수정할 파일 경로 목록. 이 목록이 Slice 간에 겹치면 병렬 실행 시 충돌이 생기므로, 겹치는 경우 Depends on으로 순차 실행을 강제한다.
- **Contract**: 이 Slice가 구현할 부분 (Contracts 섹션에서 해당 부분).
- **Depends on**: 의존하는 다른 Slice 번호. 없으면 `none`.

Depends on을 결정하는 기준:
- 같은 파일을 수정하는 Slice → 반드시 의존 관계 명시
- A의 산출물을 B가 사용 → B는 A에 의존
- 의존 관계가 없는 Slice들 → 병렬 실행 대상

### Verification List 섹션

spec의 Given/When/Then 시나리오를 구체적 검증 항목으로 변환한다.
각 항목에 해당 Slice 번호를 매핑한다. `/sd-plan-dev`가 각 Slice의 Agent에게 해당 검증 항목만 전달하기 때문이다.

검증 항목은 반드시 spec의 GWT 시나리오에서 도출한 **동작 검증**이어야 한다. "파일이 존재하는가", "섹션이 있는가" 같은 구조 검증이 아니라, "이 입력을 주면 이 동작을 하는가"를 검증한다.

예시:
- 코드: `canCancel("배송중인 주문")` 호출 → `false` 반환
- 스킬: "하드코딩된 비밀번호가 포함된 PR diff"를 입력 → "보안/Critical" 체크리스트 항목 출력

순서는 단순한 것부터 복잡한 것으로 정렬한다. 점진적으로 구현을 성장시키는 것이 한 번에 복잡한 것을 만드는 것보다 안정적이다.

### Mikado Graph 섹션 (선택)

기존 파일의 대규모 변경이 필요한 경우에만 작성한다. 변경 의존성을 DAG로 표현하여, leaf(말단) 변경부터 안전하게 적용할 수 있는 순서를 제시한다.
