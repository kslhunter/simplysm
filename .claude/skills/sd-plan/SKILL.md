# sd-plan: 구현계획서 작성

입력 문서(spec, audit, debug 등)를 기반으로 TDD 방식의 세부 구현 계획을 수립한다.

## 1. 인자 파싱

`$ARGUMENTS`를 분석하여 호출 방식을 결정한다.

### 1-1. 입력 문서 파일 경로 (+ 선택적 R번호)

`$ARGUMENTS`에 `.md`로 끝나는 경로가 포함된 경우:

- 해당 파일을 Read한다
- 경로 뒤에 `R숫자` 또는 숫자만 있으면 해당 R항목만 대상으로 한다
    - 예: `.tasks/260314143000_example/spec.md R2` → R2만 대상
    - 예: `.tasks/260314143000_example/spec.md 3` → R3만 대상
- 경로만 있으면 전체 요구사항을 대상으로 한다

### 1-2. 인자 없음

`$ARGUMENTS`가 비어있으면:

- 현재 대화 맥락에서 입력 문서 내용을 파악한다
- 대화에도 입력 문서가 없으면 AskUserQuestion으로 입력 문서 경로 또는 요구사항을 요청한다

### 1-3. 간단한 요구사항 텍스트

위 조건에 해당하지 않으면 (`.md` 경로가 아닌 일반 텍스트):

- 입력 문서 없이 해당 텍스트를 요구사항으로 직접 사용한다
- 이 경우 입력 문서 참조 없이 바로 plan을 수립한다

## 참조 문서 수정 (전 단계 공통)

2~5단계 진행 중 참조 spec/audit 문서에 누락, 오류, 모호한 점을 발견하면 다음 프로세스를 따른다:

1. 발견된 이슈와 제안하는 수정 내용을 텍스트로 설명한다
2. AskUserQuestion으로 수정 승인을 요청한다
3. 승인 시: spec/audit 파일을 Edit으로 최소 수정하고, 현재 단계를 계속한다
4. 거부 시: 현재 문서 내용 그대로 작업을 계속한다
5. 수정 범위가 과도하면 (새 R항목 추가, 요구사항 전면 재정의 등): `/sd-spec` 재실행을 안내하고 현재 plan 작성을 중단한다

**최소 수정 원칙:**
- 발견된 이슈에 직접 관련된 부분만 수정한다
- 기존 문서의 구조와 다른 항목은 유지한다
- 한 번에 하나의 이슈만 처리한다 (여러 이슈 발견 시 순차 처리)

## 2. 요구사항 분석 및 작업 계획

요구사항을 독립적 기능 단위로 작업을 분리하고, 각 작업의 유형을 분류하여 테스트 전략을 결정한다.

- 각 작업은 하나의 독립적인 TDD 사이클로 처리할 수 있는 크기여야 한다
- 작업 간 의존성이 있으면 의존성을 명시한다

**테스트 전략 결정:**

- `.claude/skills/sd-tdd/SKILL.md`를 Read하여 "작업 유형 분류" 테이블과 "테스트 시나리오 작성 원칙" 섹션을 참조한다
- 참조한 내용을 기반으로 각 작업의 테스트 전략을 결정하고 RED/GREEN Phase를 계획한다

## 3. 첨부파일 수집

대화 중 등장한 파일을 수집하여 보존한다.
첨부파일이 없으면 이 단계를 건너뛴다.

**대상 파일:**

- 사용자가 직접 첨부한 파일 (이미지, 문서 등)
- 스킬(sd-document, sd-email-analyze 등)이 분석 과정에서 추출/생성한 파일 (추출 이미지 등)

**처리:**

1. plan 저장 디렉토리의 `attachments/` 폴더를 생성한다
2. 대상 파일을 `attachments/`로 복사한다
3. 후속 초안 작성 단계에서 적절한 위치에 링크를 배치한다:
    - 이미지: `![{파일명}](attachments/{파일명})`
    - 기타 파일: `[{파일명}](attachments/{파일명})`
    - 배치 위치: 관련 작업 근처

## 4. 구현계획서 초안 작성

분석 결과를 바탕으로 아래 고정 구조의 구현계획서 초안을 작성한다.
불명확한 부분이 있더라도 최선의 판단으로 초안을 완성한다.

```markdown
# 구현계획서: {제목}

**참조 문서:** `{참조 문서 경로}` {R번호}
**대상:** `{대상 파일/패키지 경로}` {신규 작성/수정}
**작업 유형:** {유형} → {테스트 전략 요약}

## 작업 분석

{요구사항을 기능 단위로 분리한 결과}

## TDD 계획

### RED Phase: {테스트 파일명} 작성 및 실패 확인

{테스트 시나리오 목록}

### GREEN Phase: {구현 대상} 작성

{구현 계획 상세}

구현 후 테스트 재실행하여 통과 확인

### Refactor Phase

`/sd-simplify` 하여 코드 리팩터링 → 완료 후 멈추지 말고 즉시 테스트 재실행하여 통과 확인
```

**참조 문서 표기 규칙:**

- 입력 문서 파일 경로로 호출된 경우: `**참조 문서:** {경로} {R번호}` 형식으로 상단에 명시
- 현재 대화의 입력 문서인 경우: `**참조 문서:** 현재 대화의 입력 문서` 로 명시
- 입력 문서 없이 호출된 경우: `**참조 문서:**` 줄 생략

**다중 작업인 경우:**

- 작업별로 `## 작업 1: {제목}`, `## 작업 2: {제목}` ... 으로 구분하고 각 작업 내에 작업 유형, TDD 계획 포함
- 작업 간 의존성이 있으면 `## 작업 의존성` 섹션에 명시

## 5. 초안 기반 검토 질문

초안은 사용자에게 출력하지 않는다 (최종본이 파일로 저장되므로 대화창 출력은 불필요).
불명확하거나 확인이 필요한 부분이 있으면 다음 프로세스를 반복한다:

1. 초안의 특정 부분을 인용하며 **자세한 설명을 텍스트로 먼저 출력**한다
    - "초안에서 X를 Y로 계획했는데, Z 방식도 고려할 수 있습니다" 형식으로 구체적으로 질문한다
    - 선택지 각각의 장단점, 트레이드오프를 설명한다
2. AskUserQuestion을 **1개만** 호출한다 (한번에 하나씩만 질문)
3. 답변을 반영하여 초안을 수정한다
4. 추가 확인이 필요하면 1번으로 돌아간다

불명확한 점이 없거나 충분히 확인되면 질문을 종료하고 파일 저장으로 넘어간다.

## 6. 파일 저장

- 입력 문서 파일 기반 호출: 해당 문서와 동일 디렉토리에 저장. 전체면 `plan.md`, 특정 R항목이면 `{R번호}_plan.md` (예:
  `R2_plan.md`). 입력 문서가 서브디렉토리(`{R번호}/spec.md`)에 있는 경우에도 동일 규칙 적용 (해당 서브디렉토리에 저장)
- 현재 대화의 입력 문서 기반이고 파일 경로가 있는 경우: 해당 문서와 동일 디렉토리에 저장. 파일명 규칙 동일
- 입력 문서 없이 호출된 경우: 새 디렉토리 `.tasks/{yyMMddHHmmss}_{topic}/plan.md` 생성 (`yyMMddHHmmss`는 Bash
  `date +%y%m%d%H%M%S`, `{topic}`은 영문 kebab-case)

## 7. 완료 안내

문서 작성이 완료되면 다음을 출력한다:

- 완성된 문서의 파일 경로
- 사용자에게 직접 문서를 확인할 것을 권장
- 다음 단계 안내:
  ```
  /sd-plan-dev {plan경로}
  ```
