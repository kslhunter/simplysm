# sd-plan-dev: 계획서 기반 구현

이 스킬의 목적은 구현 계획서(plan.md)를 입력받아 **산출물을 실제로 작성하는 것**이다.
plan.md는 이전 단계(`/sd-plan`)에서 충분히 검토된 것으로 간주하고, 사용자에게 추가 질문을 하지 않는다.
plan.md에 명시된 대로 구현하는 것이 이 스킬의 전부이다.

한국어로 응답한다.

## 전체 프로세스 맥락

이 스킬은 3단계 파이프라인의 마지막이다:
1. `/sd-spec` → 요구 명세서(spec.md)
2. `/sd-plan` → 구현 계획서(plan.md)
3. **`/sd-plan-dev`** (이 스킬) → 구현

plan.md의 Contracts가 산출물 구조를, Slices가 작업 분할을, Verification List가 검증 기준을 제공한다.
이 세 가지를 조합하면 각 Slice를 독립적인 Agent가 검증 우선 방식으로 구현할 수 있다.

## 입력

`$ARGUMENTS`로 plan.md 파일 경로를 받는다.

## 프로세스

### 1단계: plan.md 파싱

plan.md를 읽고 다음을 추출한다:
- **Contracts**: 산출물의 구조 정의
- **Slices**: 각 Slice의 Files, Contract, Depends on
- **Verification List**: 각 검증 항목과 매핑된 Slice 번호

### 2단계: 실행 그래프 구성

**왜 실행 그래프가 필요한가**: 독립 Slice는 병렬로 실행하면 빠르지만, 같은 파일을 수정하는 Slice가 동시에 실행되면 충돌이 생긴다. 실행 그래프는 "안전하게 병렬 실행할 수 있는 Slice 그룹"을 결정한다.

구성 방법:
1. 각 Slice의 `Depends on`을 파싱한다.
2. `Files` 목록이 겹치는 Slice가 있으면, 명시적 의존성이 없더라도 순차 실행으로 표시한다.
3. 실행 그룹을 구성한다:
   - **Group 1**: `Depends on: none`이고 Files 충돌이 없는 Slice들 → 병렬 실행
   - **Group 2**: Group 1에 의존하는 Slice들 → Group 1 완료 후 실행
   - 이하 동일 패턴

실행 그래프를 사용자에게 표시한다:
```
Group 1 (병렬): Slice 1, Slice 2
Group 2 (Group 1 완료 후): Slice 3
```

### 3단계: 그룹별 실행

각 그룹을 순서대로 실행한다.

#### 병렬 실행 (같은 Group 내 독립 Slice들)

Group 내에 Slice가 2개 이상이면, **Agent tool**로 병렬 spawning한다.
각 Agent에게 해당 Slice를 구현하기 위한 모든 정보를 전달한다 (아래 "Agent에게 전달할 정보" 참조).

Group 내 Slice가 1개면 Agent 없이 직접 구현한다.

#### 순차 실행 (의존성이 있는 Group)

이전 Group의 모든 Agent가 완료된 후 다음 Group을 시작한다.
이전 Slice에서 생성된 산출물을 읽고 활용한다.

### 4단계: 검증 우선 구현 (Verify-First)

**왜 검증을 먼저 하는가**: 검증 기준을 먼저 확립하면 "이것을 통과시켜라"가 LLM의 명확한 성공 기준이 된다. 목표가 모호하면 LLM이 drift하기 쉽지만, 구체적인 검증 기준이 있으면 객관적 목표가 생긴다.

각 Slice를 다음 순서로 구현한다:

1. **Red** — Verification List에서 해당 Slice의 항목을 하나 꺼내, 이 검증이 실패하는 상태를 확인한다. 코드라면 실패하는 테스트를 작성하고 실행한다.
2. **Green** — 검증을 통과시키는 최소한의 구현을 작성하고, 통과를 확인한다. 아직 구현되지 않은 외부 의존성은 mock/stub으로 대체한다.
3. **Refactor** — 검증이 통과하는 상태에서 다음 체크리스트를 순서대로 확인한다. 해당 없으면 넘어간다.
   1. 통과시키려고 하드코딩하거나 상수를 반환한 것이 있으면 일반화한다.
   2. 같은 로직이 3회 이상 반복되면 공통 요소로 추출한다. 2회는 허용한다.
   3. 메서드/섹션이 길거나 추상화 수준이 섞여 있으면 분리한다.
   4. 이름(변수, 함수, 파일 등)이 의도를 드러내지 않으면 변경한다.
   5. 건드린 주변에 죽은 코드, 미사용 import, 불명확한 이름이 있으면 정리한다.
   6. 제거해도 검증이 통과하는 코드가 있으면 삭제한다.
   7. 검증이 여전히 통과하는지 확인한다. 실패하면 되돌린다.
4. **Repeat** — Verification List의 다음 항목으로 돌아가 1부터 반복한다.

### 5단계: 최종 검증

모든 Slice 구현이 완료되면:
1. **전체 검증 실행**: Verification List의 모든 항목을 한번에 실행하여 통과를 확인한다.
2. 결과를 사용자에게 보고한다.

## Agent에게 전달할 정보

Agent를 spawning할 때, 해당 Slice를 독립적으로 구현할 수 있는 모든 정보를 포함한다.
Agent는 plan.md 전체를 읽지 않으므로, 필요한 부분을 발췌하여 전달해야 한다.

전달할 내용:
1. **Contract** — plan.md의 Contracts 섹션에서 해당 Slice에 관련된 산출물 구조 정의
2. **Files** — 생성하거나 수정할 파일 경로 목록
3. **Verification List** — 해당 Slice에 매핑된 검증 항목들
4. **구현 방식** — Red → Green → Refactor → Repeat

Agent 프롬프트 예시:

```
다음 구현 계획에 따라 검증 우선(Verify-First)으로 구현하라.

## 산출물 구조
{Contracts에서 해당 부분}

## 대상 파일
{Files 목록}

## 검증 목록 (이 순서대로 구현)
{Verification List 항목들}

## 구현 방법 (검증 항목마다 반복)
1. **Red**: 검증 항목이 실패하는 상태를 확인한다.
2. **Green**: 해당 검증을 통과시키는 최소한의 구현을 작성하고, 통과를 확인한다.
3. **Refactor**: 다음을 순서대로 확인하고 해당되면 수정한다:
   - 하드코딩/상수 반환 → 일반화
   - 3회 이상 중복 → 추출
   - 긴 메서드/추상화 혼재 → 분리
   - 불명확한 이름 → 변경
   - 주변 죽은 코드 → 정리
   - 불필요한 코드 → 삭제
   - 검증 여전히 통과하는지 확인
4. 다음 검증 항목으로 이동하여 반복한다.

아직 존재하지 않는 외부 의존성은 mock/stub으로 대체한다.
모든 검증이 통과하면 완료를 보고한다.
```
